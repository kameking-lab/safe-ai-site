## 2. 取得方法の選択肢

### A. e-Gov 法令 API（推奨）

- v2 API: https://laws.e-gov.go.jp/api/2/swagger-ui/
- 取得手順:
  1. `GET /lawdata/{lawId}` で XML 全文取得
  2. XML から各条文の articleNum/articleTitle を抽出（条文本文は要約 or 省略）
  3. `LawArticle` 構造へ整形
  4. `web/src/data/laws/<file>.ts` として保存
- 利点: 安定 URL、機械可読、月次バッチ可能
- 欠点: 通達/告示は対象外（一部告示のみ収録）
- 実装場所: `scripts/etl/`（既存）または `scripts/law-data-import/fetch-egov.mjs`（新規）

### B. 手動 CSV/TSV（短期立ち上げ向け）

- オペレータが e-Gov の各条文を目視で確認、CSV に articleNum/articleTitle/summary を記入
- 本ディレクトリ `manifests/<lawId>.tsv` をテンプレートとして配布
- 利点: 創作リスクゼロ、優先条文だけ最短で収録
- 欠点: 工数が法令あたり 1-3 時間

### C. PDF 抽出（告示・通達向け）

- 厚労省 PDF を pdf-parse / python-docx で抽出
- 通達の場合は基発番号と本文先頭を抽出
- 利点: 告示・通達もカバー
- 欠点: レイアウト依存、要前処理

## 3. 推奨フロー

1. **Phase 1b-1**: 優先度 A 4 法令（建リ法・廃掃法・大防法・改善基準告示）を方法 A で取得
2. **Phase 1b-2**: 優先度 B 4 法令（水濁法・振動規制法・ダイ特法・貨運法）を方法 A or B
3. **Phase 1b-3**: 推進者選任基準を方法 B または PDF 抽出
4. **Phase 1c**: 優先度 C を順次

各 Phase 完了時:
- `npm run test` 通過確認
- `npm run eval:chatbot` で main/fresh Recall@5 が degrade していないこと確認
- 構造化条文 DB（article-registry）に自動取り込まれることを `npm run test src/lib/article-registry.test.ts` で確認

## 4. 受け入れ基準（各法令あたり）

- [ ] `web/src/data/laws/<file>.ts` が `LawArticle[]` の型で追加されている
- [ ] `web/src/data/laws/index.ts` の `allLawArticles` に追加されている
- [ ] `web/src/data/laws/law-metadata.ts` に対応エントリが追加されている（eGovUrl, auditedAt 含む）
- [ ] ファイル先頭コメントに「独自要約（逐語転載なし）」と明記
- [ ] 各 article の text は 1-2 文に要約（条文本文の逐語コピーは禁止）
- [ ] sourceUrl は eGovUrl で参照可能（フラグメントは article-registry が自動付与）

## 5. 禁止事項（再掲）

- e-Gov 法令本文の逐語転載は禁止（要約のみ）
- 訓練データ記憶での条文番号・タイトルの「推測」追加は禁止
- 取得元 URL が原典で照合できない条文の追加は禁止
- 既存 51 法令データの破壊は禁止
