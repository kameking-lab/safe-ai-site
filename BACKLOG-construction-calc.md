# BACKLOG-construction-calc — 建設計算コーナー 量産キュー（2026-07-12〜）

初弾（玉掛けワイヤ・単管足場・掘削勾配）の方式確立後の量産キュー。
**1計算機 = 1タスク = 1PR**。共通スキーマ準拠で Sonnet 量産可能な形にしてある。

- 設計正本: `docs/construction-calc-design-2026-07-12.md`
- 競合分析・スコアカード: `docs/construction-calc-competitive-2026-07-12.md`
- 優先度は「競合空白 × 現場需要 × 法令根拠の確立度」順。**根拠数値が公的資料で確認できない計算機は実装しない**（出典未確認の数値を載せない）。

## 量産の型（全タスク共通の完了条件）

1. `web/src/lib/construction-calc/calculators/<slug>.ts` に `ConstructionCalculator` を宣言的に定義
   （fields=プルダウン/数値+単位+範囲+既定値、basis=条文+e-Gov+法令ナビ、cautions、examples、keywords、compute=決定論）
2. `<slug>.test.ts` で数値固定（期待値は条文・公的資料から独立に手計算）＋境界値＋入力正規化
3. `registry.ts` へ登録（ハブ・[slug]ページ・AI入口・sitemap は自動追従）
4. features-catalog（construction-calc カテゴリ）へ収載＋スクショSVG 2枚（`web/public/screenshots/<slug>-{desktop,mobile}.svg`）
5. registry.test.ts の整合性ゲート全緑（lawNaviPath を足す場合は突合表へ追加）
6. ゲート: tsc 0 / lint errors 0 / vitest 全pass / build 成功。モバイル390pxスクショ確認
7. 計算はAI禁止（絶対原則）。免責 CALC_DISCLAIMER は部品側で自動表示される

## 運用メモ
- AI入口の精度が計算機の増加で劣化しないか: 10本を超えたら manifest のトークン量と誤ルーティング率を確認し、必要なら2段階ルーティング（カテゴリ→計算機）へ。
- スコアカード（competitive §3）は四半期ごとに再計測し、全計算機9点以上を維持。
