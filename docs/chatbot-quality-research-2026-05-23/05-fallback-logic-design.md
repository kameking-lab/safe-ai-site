# 「最も近い条文」提示ロジック設計

- 作成日: 2026-05-23
- 目的: 該当条文がない場合に「最も近い条文」を提示する Fallback ロジック
- 関連: 04-hallucination-prevention-design.md セクション4

## 1. 設計原則

社長要求: 「該当条文が無くても最も近い条文を提示、条文番号は絶対間違えない」

原則:
- 「該当条文なし」と冷たく突き放さない
- ただし「最も近い」と称して無関係な条文を出さない
- 関連度に応じて段階的に提示し、関連の弱さを明示する
- すべての提示条文は必ず構造化条文 DB に存在する実在条文

## 2. 3段階分岐

### 2.1 完全一致(Direct Hit)

定義:
- normalizedScore >= 0.75 かつ relevantArticles >= 2
- または PIN が刺さって主条文 + 関連条文セットが揃っている

応答スタイル:
- 結論を本文先頭に断定形で1-2文(現状 SYSTEM_PROMPT が指示済)
- 条文番号を「○○則第XX条(施行：YYYY年MM月、所管：厚生労働省)」3点セット
- 「明確な規定は見つかりませんでした」等のぼかし禁止(現状ルール8-9)

例(Direct Hit):
```
質問: 「フォークリフトの運転に必要な資格は?」

回答: 最大荷重1t以上のフォークリフトの運転には「フォークリフト運転技能講習」の修了が必要です(労働安全衛生法 第61条第1項、労働安全衛生法施行令 第20条第11号)。 1t未満は特別教育(安衛則 第36条第5号)で足ります。
```

### 2.2 部分一致(Adjacent Hit)

定義:
- 0.5 <= normalizedScore < 0.75 または relevantArticles == 1
- 主条文は存在するが、質問の論点全体をカバーしない

応答スタイル:
- 「ご質問の△△については、関連する一般条項として□□があります」
- 提示する条文を「直接答える条文」「関連する一般条項」と明示区別
- 不足箇所は「本ツールの提供データには明示的な規定が見つかりませんでした」と書く(現状ルール9に整合)

例(Adjacent Hit):
```
質問: 「夜勤中に利用者から暴力を受けたが、安全配慮義務違反になるか?」

回答: 「夜勤中の暴力対応」に直接該当する条文は安衛法・関連省令には明示されていません。 ただし関連する一般条項として、労働契約法 第5条(使用者の安全配慮義務)があり、使用者は労働者の生命・身体の安全を確保する義務を負います(労働契約法 第5条)。 また、安衛法 第59条(雇入れ時等の安全衛生教育)により、想定リスクへの教育が義務付けられています。

具体的な対応手順については、安衛則・労働基準法本体には明示規定がありません。 厚生労働省「カスタマーハラスメント対策企業マニュアル」等の指針をご参照ください。
```

### 2.3 完全不一致(Out-of-Scope)

定義:
- normalizedScore < 0.5
- PIN も非該当
- 関連通達・MLIT資料も非該当

応答スタイル:
- 「本ツールの提供データ範囲(50法令 + 関連通達)では確証ある回答が得られませんでした」を明示
- 関連分野として該当しそうな法令カテゴリを2-3件提示(他法令誘導)
- e-Gov 法令検索 + 専門家相談を案内
- scopeWarnings に出力

例(Out-of-Scope):
```
質問: 「ドローンを建設現場で使う場合の事前手続は?」

回答: ⚠️ この質問は本ツールの提供データ(50法令 + 関連通達)の主要範囲外です。

関連分野としては:
- 航空法 第132条(無人航空機の飛行)— 国土交通省所管
- 国交省「無人航空機の飛行に関する許可・承認」ガイドライン
- 建設業労働災害防止規程(建災防)の関連条文(高所作業との関係)

詳細は国土交通省の無人航空機運航管理サイトおよび e-Gov 法令検索(https://laws.e-gov.go.jp/)でご確認ください。 個別事案の判断は労働安全コンサルタント等の専門家にご相談ください。
```

## 3. 実装フロー

### 3.1 既存実装の修正

現状(route.ts:310-349): normalizedScore < 0.5 でテンプレート応答を返す。 改修して「最も近い条文」提示ロジックを挿入:

```
if normalizedScore < 0.5:
  // Layer 3 Fallback
  partialHits = searchPartialMatches(message)  // 関連法令カテゴリ判定
  if partialHits.length > 0:
    → "Out-of-Scope but Adjacent" 応答(関連法令誘導)
  else:
    → "Out-of-Scope" 応答(e-Gov 誘導のみ)
else if normalizedScore < 0.75 or relevantArticles.length == 1:
  → "Adjacent Hit" 応答(主条文 + 明示規定なしの旨)
else:
  → "Direct Hit" 応答(現状フロー)
```

### 3.2 searchPartialMatches の実装

質問キーワードから関連法令カテゴリを判定する補助関数:

```ts
const TOPIC_TO_LAW_CATEGORY = {
  「ドローン」: ["航空法", "建設業法"],
  「アルコール検知」: ["貨物自動車運送事業法", "道交法"],
  「カスハラ」: ["労働施策総合推進法 第30条の2", "労働契約法 第5条"],
  「夜勤暴力」: ["労働契約法 第5条", "安衛法 第59条"],
  「適性診断」: ["貨物自動車運送事業法"],
  「感染症就業制限」: ["感染症法", "安衛則 第61条"],
  「土砂崩壊前兆」: ["安衛則 第361条", "建災防規程"],
  ...
}

function searchPartialMatches(query): RelatedLawSuggestion[] {
  for topic, candidates in TOPIC_TO_LAW_CATEGORY:
    if query.includes(topic):
      return candidates.map(...)
  return []
}
```

このマッピングは PINNED_TOPICS と類似だが、「該当条文が本ツールに含まれない場合」用。

## 4. 既存機能との統合

### 4.1 suggestRelatedLaws(現状)

chatbot-enrichment.ts:165-214 に既に suggestRelatedLaws があり、RELATED_LAW_GROUPS に基づき関連法令を最大4件サジェスト。

これは「ヒット時の補足」用。 Fallback では別の関数(suggestFallbackLaws)を用意する。

### 4.2 suggestDigDeeperLinks(現状)

chatbot-enrichment.ts:219-289 で事故事例・通達・業種別レポート・法令検索動線を返す。

Fallback でも有用なので維持。 ただし「該当データ範囲外」用に専用 dig deeper を追加(例: e-Gov 検索リンク、専門家相談 CTA)

### 4.3 mhlw-notices の通達検索

route.ts:230 で searchRelevantNotices が呼ばれており、関連通達があれば Fallback でも 提示。

「条文ヒットなし、通達のみヒット」のケースは Layer 3 で:
- 「該当する法令本則の規定はありませんが、関連通達があります」
- bindingLevel(binding/indirect/reference)を明示

## 5. UI 表示の改善案

### 5.1 部分一致時の視覚区別

- 「直接該当」: 通常表示
- 「関連条項」: ✏️ アイコン + 「関連する一般条項」見出し
- 「データ範囲外」: ⚠️ アイコン + 「本ツール範囲外、e-Gov 推奨」見出し

### 5.2 信頼度の明示

- 既存 confidenceScore(0-1)を UI で「信頼度: high/medium/low」表示
- Fallback 時は「直接該当条文なし(関連条文を参照中)」表示

### 5.3 dig deeper の差別化

- Direct Hit: 通常の suggestDigDeeperLinks
- Adjacent Hit: 「他の関連条文も見る」「業種別事故事例」「通達カテゴリ」
- Out-of-Scope: 「e-Gov 法令検索」「専門家に相談」「フィードバック送信」(個人運営研究プロジェクト体裁のため「無料相談」は使わない)

## 6. 想定精度

- Fallback 実装後の想定:
  - 「明確な規定なし」回答の質改善: 関連条文を必ず1件以上提示
  - ペルソナテスト失敗11件のうち、5-7件が △ → ◯ に改善見込み
  - 「条文番号は絶対間違えない」保証は維持(提示する全条文は構造化条文 DB に存在する実在条文)

## 7. 実装規模

- searchPartialMatches + fallback-law-suggestions.ts: 1-2日
- route.ts の Fallback 分岐改修: 0.5日
- UI 表示の信頼度・分岐表示: 1-2日
- テストケース追加(ペルソナテスト失敗11件): 0.5日
- 合計: 3-5日

## 8. 評価メトリクス

- Fallback Quality(完全一致なし時に妥当な代替を提示できる率): 80%+ を目標
- False Adjacent(直接該当を Adjacent と誤判定): 5% 未満
- False Direct(関連条項のみなのに Direct と誤判定): 2% 未満
- User-perceived Helpfulness(主観評価): 別途ユーザーテストで計測

## 9. リスクと対策

- リスク1: Adjacent Hit の判定基準(0.5-0.75)が経験的で、ベンチで最適化が必要
  - 対策: rag-100q.fixture.ts に「期待結果 = Direct/Adjacent/Out」のラベルを追加し、最適閾値をグリッドサーチ
- リスク2: 関連法令誘導が外部法令(e-Gov)を勧めることで、本ツールの価値が薄まる
  - 対策: 関連法令誘導は補助情報、主応答は「本ツールでわかること」を最優先で提示
- リスク3: TOPIC_TO_LAW_CATEGORY のメンテナンス負担
  - 対策: 失敗ログ収集 → 月次でレビュー追加、というオペレーションを設計
