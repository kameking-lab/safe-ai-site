# オーナー判断必須事項 一覧

- 作成日: 2026-05-23
- 対象: チャットボット品質改修の着手前に必要なオーナー判断
- 形式: 判断項目ごとに「現状」「論点」「選択肢」「推奨」を提示

## D1. 建設業振り切りの程度

### 現状
- 50法令体制(33基幹 + 拡張 + 施行令類)
- UI コピーは「33法令以上」(ChatbotBody.tsx)
- 建設業向け振り切り方針が確定(社長要求)

### 論点
- 33→50→「+建設業重視で60法令+α」のどこまで広げるか
- 安衛則の足場・掘削・型枠支保工の収録範囲を「主要条文のみ」とするか「全条文」とするか
- 運送業・小売業・医療福祉ペルソナの困りごとへの対応をどこまで切るか

### 選択肢
- A: 現状50法令を維持し、建設業関連の安衛則条文(足場・掘削・型枠等)を全条文化(+200-300条文程度)
- B: 50法令 + 貨物自動車運送事業法・道交法・カスハラ指針等の枠外法令を追加(50→60法令程度)
- C: A + B 両方(建設業特化を強化しつつ、運送・医療福祉ペルソナも一部対応)

### 推奨
- 推奨: A
- 理由: 「建設業向け振り切り」方針に最も整合。 運送業・小売業ペルソナは Phase 4 以降または別機能(/heat-illness-prevention 等)で対応可。 33法令枠外法令の追加はコーパス膨張で誤マッチを誘発するリスクあり

## D2. 通達/告示原文URLの応答内表示形式

### 現状
- 応答末尾に【関連通達・告示】見出しで列挙(現状)
- bindingLevel ラベル付き

### 論点
- 通達カードコンポーネント(NoticeCard)を新設するか、現状のテキスト列挙のままにするか
- インライン展開 vs 折りたたみ vs 別ページの3択

### 選択肢
- A: インライン展開(最大3件まで、それ以上は折りたたみ)
- B: 常に折りたたみ(クリックで展開)
- C: 別ページ(/chatbot/notices?q=... のような専用ビュー)

### 推奨
- 推奨: A(インライン展開、最大3件まで、それ以上は折りたたみ)
- 理由: スマホ片手間ユースケース(建設業職長ペルソナA)では即座に見える方が良い。 ただし通達5件以上は折りたたみで画面圧迫を避ける

## D3. e-Gov 法令API v2 の定期取込み実装

### 現状
- 法令データは静的(web/src/data/laws/*.ts)
- 既存スクリプトに e-Gov API 直接連携は未確認(parse-mhlw-preliminary.py 等は別ソース)

### 論点
- 月次バッチで e-Gov API から法令本文を取得 → 構造化条文DBを自動更新するか
- 静的データのまま手動更新を続けるか

### 選択肢
- A: 月次バッチで e-Gov API から自動取得 + 差分検出 + 自動 PR 作成
- B: 半年ごとに手動更新
- C: 改正検知のみ自動化(本文取得は手動)

### 推奨
- 推奨: A(ただし Phase 1-4 完了後の追加施策として実装、本研究の Phase 範囲外)
- 理由: 法令改正は年数回あり、手動更新では追従が困難。 ただし初期実装は静的データのままで OK、Phase 5 として e-Gov 取込みを設計

## D4. Gemini Flash → Pro 切替の是非

### 現状
- gemini-2.5-flash を使用(route.ts:359)
- Pre-generation 制約を強化すると Flash の追従能力に限界がある可能性

### 論点
- Gemini Pro に切替で品質向上が期待できるが、コスト・応答時間が悪化
- 現状 Flash で Citation Accuracy@1 = 99.5%+ が達成可能か未検証

### 選択肢
- A: 引き続き Flash 使用、Layer 2 で誤りを補正
- B: Pro に切替
- C: ハイブリッド(高信頼質問は Flash、低信頼や複雑質問は Pro)

### 推奨
- 推奨: A → Phase 2 完了後に再評価
- 理由: Phase 2 の Post-generation 照合が機能すれば Flash で十分な可能性が高い。 Pro 切替は最後の手段

## D5. 通達収録拡大と「個人運営研究プロジェクト」体裁の整合(PR #182)

### 現状
- 通達 1,069 件収録
- PR #182 で「個人運営研究プロジェクト」体裁に方針転換済
- 「専門家チーム」→「労働安全衛生コンサルタント(登録番号260022)」表記
- 「無料相談」CTA →「ご意見・改善提案を送る」に変更済

### 論点
- 通達 1,069 → 1,500-2,000 件に拡大した場合、「研究プロジェクトの範囲を逸脱して商用サービス化している」と見なされるか
- 著作権上の問題はないが、対外的な体裁の整合性

### 選択肢
- A: 通達拡大可、ただし「個人の研究プロジェクトが公開している学習用資料」として明示
- B: 通達拡大は1,500件まで、それ以上は商用化検討時に拡大
- C: 通達拡大せず、URL添付のみで対応(本文・要約は持たない)

### 推奨
- 推奨: A
- 理由: 通達自体は政府標準利用規約で再配布可。 本文要約は持たず、構造化メタデータ + 原文URLのみ保持なら体裁問題なし。 ただし「商用サービス」表記は厳禁、教材公開の文脈を維持

## D6. ハルシネーション Pre-gen 制約に違反した応答の扱い

### 現状
- 違反検出 → 警告追記 + scopeWarnings(現状の detectOutOfScopeLawReferences の動作)

### 論点
- Pre-gen 制約を導入後、Gemini が制約を破ったときの動作
- 「警告のみで応答返す」「自動破棄して低信頼テンプレートに置換」の2択

### 選択肢
- A: 警告のみ(現状動作の延長、UX 維持)
- B: 自動破棄(信頼度 high → low に降格、e-Gov 誘導テンプレート)
- C: 段階的(初期は警告のみ、ベンチで違反率を計測してから自動破棄に移行)

### 推奨
- 推奨: C(段階的)
- 理由: 初期動作は既存 UX を維持しつつ、Layer 2 のメトリクスで違反率を可視化。 違反率が 1% 以下に安定すれば自動破棄に切替。 動的に切替可能なフラグ(環境変数 HALLUCINATION_STRICT_MODE 等)で制御

## D7. サブスク課金との結合

### 現状
- NEXT_PUBLIC_PAID_MODE = false(課金機能フラグ)
- Stripe Price ID(STRIPE_PRICE_PRO/PREMIUM)準備済
- 課金実装は未着手(オーナー確認必須事項として CLAUDE.md に記載)

### 論点
- 本研究の品質改修が完了したチャットボットを「無料 vs 有料」のどちらに位置付けるか
- 通達 URL 添付、Fallback、ハルシネーション絶滅機能を全部無料で提供するか

### 選択肢
- A: 全機能無料(現状方針、研究プロジェクト体裁を維持)
- B: 基本無料 + プロ機能(通達 URL 添付、Fallback の詳細表示)を有料化
- C: 基本無料 + 質問数制限(月100問まで無料、超過は有料)

### 推奨
- 推奨: A(本研究の範囲では)
- 理由: 「個人運営研究プロジェクト」体裁との整合。 課金化は別途独立した意思決定として扱う(本研究の Phase 範囲外)

- Slack/メール/対面でも可
