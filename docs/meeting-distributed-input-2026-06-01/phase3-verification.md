## セキュリティ検証（認可の穴＝他人の現場データ漏洩。全て通過）
- **トークン越境不可**: M1のトークンで入った投稿は、別打合せ書(M2)の取り込みに一切出ない（M1集約=2/M2集約=0、本番でもM1=1/別=0）。PASS
- **匿名直アクセス拒否**: アプリは `NEXT_PUBLIC_SUPABASE_ANON_KEY` を**どこでも使っていない**＝クライアントからDBへ触る経路が存在しない。加えて新3テーブルは **RLS有効＋anonポリシー無し**＝匿名ロールは全行拒否（owner実行のSQLで担保）。＝二重に遮断。PASS
- **無効/未知トークンの拒否**: 形式不正・存在しないトークンは 404（情報を一切返さない）。PASS
- **他社行・元請確定欄の改ざん不可**: 投稿の編集鍵 contribution_id は**サーバー生成の推測不能64hex**。他社はそれを知らず他社行を更新できない（攻撃者投稿は別の新規行＝既存C1無傷）。`sanitize` が **actualCount/appendNote/parentId/id を保存payloadから除外**（本番でも actualCount 除外を確認）＝協力会社は元請の確定欄を持ち込めない。PASS
- **元請の所有権**: 取り込みは device_id 照合＝別端末（攻撃者）はD1のM1投稿を取得できない（件数0）。PASS

## 競合（要件1）検証
- **2社同時入力でデータ消失なし**: 別 contribution_id ＝別行で双方残る（取り込み順不同でも両社）。PASS
- **同一行の楽観ロック**: 古い base での更新は 409 で拒否（黙って消さない）。PASS
- **実DBで発覚した不具合を修正**: Postgres は `submitted_at` を `…+00:00`、サーバー生成は `…Z` で返すため、文字列比較の競合判定が**正当な自社更新を全て誤って409**にしていた。`hasWriteConflict` を**時刻(ms)比較**に修正（PR内 fix コミット＋テスト追加）。修正後、更新→成功を本番で確認。PASS

## 履歴・復元（要件2）検証
- **履歴記録**: 送信のたびスナップショット追記（更新後 historyCount=2 を実DBで確認）。PASS
- **一つ前に戻す**: revert で直前内容(V1)へ復元を実DBで確認。PASS
- **期限切れ除外/自動削除（30日）**: 読み取り除外(activeHistory)＋書き込み時GC(`recorded_at<now-30d`削除)。30日経過は実DBで早送りできないため**ロジックを単体テストで担保**（期限切れ判定・除外・新しい順）。PASS（ロジック検証）
