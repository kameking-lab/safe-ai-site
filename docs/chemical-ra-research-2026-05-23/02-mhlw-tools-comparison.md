# 02: 厚労省ツール群との比較

作成日: 2026-05-23
範囲: 厚労省（MHLW）・NITE・JIS が公開する化学物質リスクアセスメント関連ツールを棚卸しし、当サイト実装の差別化機会を洗い出す。

注: 本ドキュメントは公開情報に基づく一般的な評価であり、各ツールの仕様は変更される可能性がある。本格実装時には最新の公開仕様を再確認すること。

---

## 1. 厚労省公式ツール群（要点整理）

### 1.1 CREATE-SIMPLE（クリエイト・シンプル）

提供元: 厚生労働省「職場のあんぜんサイト」
バージョン: v3.0 系列が現行（2023〜2024年更新版が流通）
形式: Excel マクロ（xlsm）／一部Webフォーム

**機能:**
- 物質名・CAS番号で物質を選択
- GHS分類自動引用（政府版GHS分類結果に依存）
- 取扱量・換気・作業時間・呼吸用保護具 → ばく露濃度の推定
- リスクレベル I-IV 判定（許容濃度との比較）
- 工学的対策・管理的対策・保護具の推奨

**カバー物質数:** 政府版GHS分類結果のうち約 3,000-3,500物質（更新で増減）

**判定軸:**
- 吸入リスク（中心）
- 経皮リスク（簡易判定）
- 危険性（火災爆発、限定的）

**限界・弱点:**
1. **Excelマクロ依存** → スマホ・タブレットで使えない、社内IT制限で動かないケース多発
2. **混合物・製品の取扱いが弱い** → 構成成分ごとに個別実行が必要
3. **業種テンプレートなし** → 塗装・解体・防水等の現場固有条件が反映できない
4. **出力固定書式** → 1枚紙に書き換え不可、PDFカスタム不可
5. **法令条文への直接連携が弱い** → 「特化則対象」と記載があっても条文番号や通達URLへの直接遷移なし
6. **更新サイクルが官公庁ペース** → GHS分類追加が年1-2回
7. **検索動線が薄い** → 物質マスタを別途調べないと使えない
8. **AI連携なし** → 自然言語入力不可

### 1.2 CSCL ツール／化学物質管理 ICT 関連

厚労省「化学物質管理に関する法律対応」を支援する各種ツール群:

- **濃度基準値・8時間基準値テーブル** = 告示177号別表のWeb版（PDF＋検索）
- **作業環境測定結果評価ツール** = 第1管理区分判定（測定機関向け）
- **個人ばく露濃度測定 評価ツール** = 個人サンプラー結果の評価

これらは「測定後の評価」が主で、「事前のRA」に直接使えるツールは CREATE-SIMPLE が中心。

### 1.3 NITE「化学物質総合情報提供システム（CHRIP）」

提供元: 製品評価技術基盤機構（NITE、独立行政法人）
URL: https://www.nite.go.jp/chem/chrip/chrip_search/

**機能:**
- 物質名・CAS番号で化学物質情報検索
- GHS分類結果（政府版）= 約 3,000-3,500 物質
- 各国規制情報（化審法、化管法、毒劇法 等）
- IARC 発がん性分類
- メタデータ豊富

**限界:**
- 「情報提供」が主で、RA 判定機能なし
- 取扱量・換気等の条件入力UIなし
- 結果は CSV/PDF ダウンロード型

### 1.4 職場のあんぜんサイト（MHLW）

URL: https://anzeninfo.mhlw.go.jp/
モデルSDS約3,000物質、災害事例、法令検索。**RAツールではない**が、当サイトの fallback データ源として参照。

### 1.5 GHS分類JISハンドブック（JIS Z 7252:2019, Z 7253:2019）

- Z 7252: 分類方法（混合物・物質）
- Z 7253: ラベル・SDS 記載項目

実装時の参照基準。ラベル・H/P文の文言は JIS にロックされる。

## 4. CREATE-SIMPLE の弱点 トップ7（当サイトの差別化機会）

1. **Excel/PCロック** → 当サイトは Web完結、スマホ・タブレット・サイネージで動作
2. **物質マスタ固定** → 当サイトは外部DB統合で 27,000規模を狙える
3. **業種テンプレ不在** → 建設業特化（塗装・解体・防水・地盤改良）で初学者でも即運用可
4. **1枚紙出力なし** → KY/日誌(#263)と同じ用紙ファースト型UIで現場運用前提
5. **法令連携が文字列** → 条数・通達番号・リーフレットURLへ直接遷移可能に
6. **混合物の手間** → 製品名→構成成分→自動RAのパイプラインを既に保有
7. **AI不在** → Gemini で解説文生成、複合判定の根拠説明、自然言語クエリ対応

## 5. 「厚労省互換モード」の必要性

社長要求の「厚労省超え」と現場運用の「行政互換」は両立すべき。設計案:

- **行政提出用 = CREATE-SIMPLE互換**: ベンチレーション3段階、取扱量3段階、I-IV判定をそのまま再現
- **詳細モード = 当サイト独自**: 流量別換気、連続/間欠、複合曝露、業種テンプレ
- 切替トグル「行政提出向け／詳細判定」を1枚紙UIに置く

これにより「行政から指摘されたら CREATE-SIMPLE と同じ結果が出る」+「現場では精度の高い詳細判定」が両立。

## 6. ベンチマークの考え方

「厚労省超え」を客観的に主張するには以下のメトリクス測定が要る（公開可能な範囲で）:

- カバー物質数（公的出典明示の物質数）
- 業種別テンプレート数
- 法令引用の精度（条文への直リンク率）
- レスポンス速度（ファースト判定の表示時間）
- モバイル運用適性（Lighthouse モバイルスコア）
- 1枚紙印刷品質（A4縦、白紙余白、本文文字サイズ、印刷時の崩れゼロ）

これらは Phase 完了時の受入基準として `08-implementation-roadmap.md` に組み込む。
