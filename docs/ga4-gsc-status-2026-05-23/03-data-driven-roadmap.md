# 実測データ取得後の SEO 深化ロードマップ

**前提:** `02-required-actions.md` 完了により GA4 ＋ GSC が稼働している状態。
**狙い:** ペルソナテストで定まった改善方針（建設業振り切り含む）の効果を実測で検証し、次手を確度高く打つ。

## フェーズ 0: ベースライン確定（稼働後 +0 〜 +14 日）

### 0.1 取得対象データの固定

GSC 側で以下を週1スナップショット（手動 export → `docs/seo-baseline-YYYY-MM-DD/`）:

- 主要クエリ TOP 50（impressions / clicks / CTR / position）
- ランディングページ TOP 50
- デバイス内訳
- 検索クエリ × ランディングページの組み合わせ TOP 30

GA4 側:

- 流入元別セッション数（organic / direct / referral / social）
- 上位 LP のセッション時間 / 直帰率
- スクロール深度 90% 到達率（カスタムイベントで取れていれば）

### 0.2 ベースライン KPI（建設業ペルソナ向け）

| 指標 | 取得元 | 初期目標 |
|------|--------|----------|
| 建設業関連クエリの impressions | GSC | 14日後 ≥ 5,000 |
| 建設業関連クエリ平均 position | GSC | 14日後 ≤ 30 |
| /accidents-reports/construction 系 LP の CTR | GSC | 14日後 ≥ 1.5% |
| 建設業 LP のセッション時間中央値 | GA4 | 14日後 ≥ 60秒 |

これらは 0.1 のスナップショット蓄積で算出可能。

## フェーズ 1: クイックウィン特定（+14 〜 +30 日）

### 1.1 「あと一歩」クエリの抽出

GSC で `position 4-20` かつ `impressions ≥ 100/週` のクエリを抽出。
タイトル / H1 / メタディスクリプションのキーワード調整で順位押し上げを狙う（CTR/position 同時改善が見込める層）。

### 1.2 高 impression 低 CTR の修正

`impressions ≥ 500/週` かつ `CTR ≤ 1%` のクエリ × LP 組み合わせを抽出。
タイトルタグの訴求改変 A/B（順位低下リスクが少ない範囲で2週間ごと差し替え→position 維持率を計測）。

### 1.3 内部リンク強化候補の特定

GA4 の「上位 LP の next_page」イベントから、回遊が切れているページを特定。
特に建設業ペルソナの動線（事故DB → KY → 法令確認）で離脱率が高いポイントを優先補強。

## フェーズ 2: コンテンツ拡張の優先順位付け（+30 〜 +60 日）

### 2.1 検索意図 × 既存記事ギャップ分析

GSC で impressions あるが LP 未整備のクエリを抽出 → ランディングページ新規作成の優先順位を実需で並べる（CLAUDE.md の「優先度の高い課題」と GSC の実需を突合）。

### 2.2 建設業振り切り戦略の効果検証

ペルソナテストで定めた改善後と改善前の同一クエリ群で：

- position 平均が改善方向に動いたか
- 建設業関連 LP のセッション時間 / コンバージョン（KY 用紙ダウンロード等）が伸びたか
- 想定外に伸びている非建設業クエリがあれば派生展開を検討

### 2.3 季節要因の補正

7日 / 30日 / 90日の比較で建設業ペルソナの検索行動の季節性（夏季：熱中症、秋季：高所作業、年度末：法改正）を把握 → 来期の準備時期を決める。

## フェーズ 3: 自動化と継続運用（+60 日 〜）

### 3.1 週次 GSC スナップショットの自動化

候補実装:

- Vercel Cron + 新規 API ルート `/api/cron/gsc-snapshot`（既存 `search-console-client.ts` 流用）
- 取得結果を Vercel Blob または `data/seo-snapshots/` に JSON 保存
- 4週分を保持してダッシュボード `/stats` の SEO セクションに「前週比 / 前月比」を表示

着手判断は **データドリブン**：手動運用を 4-8 週続けて「自動化に値する蓄積量」が確証できてから（先に作って数値が出ないより、数値が出ることを確認してから工数を払う方が無駄が少ない）。

### 3.2 アラート条件

- 主要クエリ TOP 10 のうち position が前週比 +5 以上悪化したクエリが出たら通知（Resend で `kenshi.ycc@gmail.com` へ）
- 主要 LP の clicks が前週比 -30% 以下なら通知
- 404 系の検索流入が発生したら通知

これも実装着手は最初の数値悪化が起きてから（取らぬ狸対策の前に、起こったときに対処できる手順を持つだけで初期は十分）。

## やらないと決めること

- **AB テストツール本格導入は当面しない**：実測サンプル数（建設業クエリで月 5,000 impressions 程度）では統計的有意性が出にくい。タイトル/メタの「定性的な改善 → 2-4 週運用 → GSC position 観測」サイクルで十分な期間が当面続く。
- **競合分析ツール（Ahrefs / Semrush）の導入もしない**：GSC で自サイト実需が見えれば、競合スコアより自サイトの「あと一歩クエリ」の方が ROI が高い。
- **GA4 イベント追加の網羅化もしない**：ペルソナテストで効いた指標（KY 用紙ダウンロード / 事故DB 検索 / チャットボット質問送信）だけ確実に追跡し、それ以外は追加要請があってから足す。

## 着手順序まとめ

1. オーナーが `02-required-actions.md` を実施 → 環境変数 6本登録 + 再デプロイ
2. セルフチェック手順で稼働確認
3. 稼働確認できた翌日からフェーズ 0 開始（14日のベースライン期間）
4. ベースライン取得後にフェーズ 1 のクイックウィン候補抽出を Claude に依頼
5. 60日後にフェーズ 3 の自動化検討

**最初の意思決定は「自動化を組むかどうか」ではなく「フェーズ 0 のスナップショットを定期的に取り始めるかどうか」**。スナップショットさえ蓄積していれば、後から分析方針を変えても遡って効果検証ができる。
