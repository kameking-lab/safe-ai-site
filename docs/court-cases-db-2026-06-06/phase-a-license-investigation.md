# 労災/労働判例DB（数百件・検索可能）— フェーズA: データソース／ライセンス調査

調査日: 2026-06-06 / 調査者: 自律改善ループ
目的: 現状16→23件の手作業キュレーションを、**数百件規模・カテゴリ＋自由ワード検索可能**な
判例データベースへ発展させる。本フェーズは「商用サイト（anzen-ai-portal.jp）で**合法的に**
数百件の実在労働判例を扱えるソースが本当にあるか」を厳密に確認するのが目的。
**結論を先に: ライセンス的に使える"そのまま数百件の一括データセット"は現状存在しない。
ただし判決文そのものは著作権の対象外（著作権法13条）であり、政府公開ソースからの
"再現可能なキュレーション・パイプライン"を複数サイクルで積み上げる道なら合法的に到達できる。**

## 0. 法的な土台（最重要・有利な事実）
- **著作権法第13条**: 「憲法その他の法令」「国・地方公共団体の告示・訓令・通達等」「**裁判所の判決・決定・命令**」
  「国等が作成するこれらの翻訳物・編集物」は**著作権の目的とならない**。
  → 判決文の本文・裁判要旨（国が作成した部分）は、**商用利用を含め自由に利用できる**（出典: e-Gov法令検索 著作権法13条、文化庁・各種解説）。
- ただし **(a) 取得元サイトの利用規約（スクレイピング可否等）** と、**(b) 民間の判例集出版社が付した独自の解説・編集**（こちらは著作権あり）は別問題。**13条が免除するのは"判決そのもの"と"国が作った編集物"**であって、民間DBの解説文ではない。
- 当サイトの作法（既存16→23件と同じ）= **判決文の長文転載はせず、当サイトによる要約＋出典リンク**。これは13条に頼らずとも著作権上安全（事実・短い引用＋自作要約）。大量データでもこの作法を厳守する。

## 2. 正直な結論
- **「商用利用可・そのまま数百件・一括取得できる労働判例データセット」は2026年6月時点で存在しない。**
  NIIは研究限定、法務省DBは未整備、民間DBは有償・再配布不可。
- **だが判決そのものは著作権フリー（13条）**であり、**政府公開ポータル（厚労省・中労委）＋裁判所の個別公開判例**を
  出典として、**当サイト要約＋出典リンク方式**でキュレーションするのは**完全に合法**。
- したがって数百件への道は「巨大データセットの一括投入」ではなく、**政府公開・著作権フリーのソースから、
  出典を必ず付けて、再現可能な手順で一件ずつ積み上げるキュレーション・パイプライン**。これは一朝一夕では
  終わらない＝**毎サイクル一歩ずつ積み上げる**性質の大仕事。社長の意図（コツコツ大仕事）と合致する。

## 3. 推奨アーキテクチャ（フェーズB以降の方針）
- **データ規模と保存先**: 数百件（〜1,000件程度）なら**静的データ（型付きTS/JSON＋ビルド時インデックス）で十分**。
  env/DB不要・オーナー承認不要・デプロイが軽い。クライアント全文検索も数百件なら高速。
  → **まず静的データ路線で数百件規模まで設計**。数千件を超え、サーバ全文検索や更新頻度が問題化したら
  そこで初めてSupabase（Path Aでオーナー承認・schema.sql提示）を検討する。**現段階ではDB変更は不要**。
- **スキーマ拡張**: 既存 `CourtCase` 型（id/name/court/date/issues/field/oneLine/summary/holding/practicePoints/sources）を
  踏襲しつつ、検索のため `keywords[]`（事故型・業種・条文等のタグ）、`industry`（業種）、`courtType`（最高裁/高裁/地裁）、
  `decade`（年代）を追加。数百件をカテゴリ＋自由ワードで引けるようにする。
- **検索UX**: 既存 `CourtCasesBrowser`（争点/分野/キーワード、#357でURL同期済）を、件数増に耐えるよう
  カテゴリ別件数バッジ・年代/裁判所フィルタ・ページング/件数表示へ拡張。

## 4. フェーズ計画（複数サイクル・複数日で積み上げ）
- **A（本書・完了）**: データソース／ライセンス調査。→ NII研究限定で不可、政府公開ソース＋13条で合法キュレーション路線に確定。
- **B（次の一歩）**: スキーマ拡張（keywords/industry/courtType/decade）＋検索UIの件数増耐性化＋**データ投入パイプラインの型・バリデーション**（捏造防止の構造ガードを数百件規模に拡張）。env/DB不要の静的路線で設計・実装。
- **C（反復）**: 政府公開ソース（厚労省ポータル・中労委DB・裁判所個別公開）から、**全件出典付き・実在確認済み**で
  毎サイクル十数件ずつ追加。まず数十件パイプラインを通し、検証できたら拡大。
- **D（反復）**: 数百件から目的判例へ素早く辿り着く検索体験（カテゴリ＋自由ワード＋年代/裁判所）の作り込み。
- **オーナー承認が必要になる地点**: 静的路線で限界（数千件・サーバ検索が必要）に達したら、Supabase導入をPath Aで承認依頼（schema.sql提示）。**それまではDB変更しない。**

## 5. 絶対に守る制約
- **捏造禁止**: 数百件規模でも全件、出典で実在確認。架空判例は1件もダメ。各件に出典（裁判所事件番号/政府ページURL等）を付与し追跡可能に。
- **ライセンス厳守**: 研究用途限定（NII）等は使わない。政府標準利用規約・13条で再利用可能なものだけ。
- **著作権配慮**: 判決文の長文転載はせず、要約＋出典リンク（既存と同じ作法）。
- **安全停止設計**: 静的路線なのでmainは常にデプロイ可能。1挑戦1ブランチ・CI緑でマージ。
