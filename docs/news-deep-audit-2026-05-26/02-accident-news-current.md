監査日: 2026-05-28 / 現状成熟度: **約70/100**

- 月次速報（最鮮度の構造化チャネル）: `data/accidents/monthly-sokuhou.json` = 令和8年4月速報・3月末累計、fetchedAt 2026-05-27、skipped 0
- 速報集計: `aggregates-mhlw/summary-2026-preliminary.json` = 令和8年1〜3月、死亡 Jan-Mar 114（前年142）、as_of 令和8年4月6日

- 報道feed: `news-feed/approved/index.json` = 10件、最新2026-05-27

**直近改正・速報への追従は良好。** 月次ETLは厳格ヘッダ検証＋skipped未確認明記＋diff-onlyで、月次速報ETLパターン（形式検証＋未確認明記＋推測値禁止）の手本。

## 2.2 取込頻度・速報性・表示

- 月次速報: cron 15日（月次）。**build-time JSON読み込みのため、新データはデプロイ/revalidate後に反映**。`/accidents-reports`のrevalidate 30日 → 最鮮度の公式数値が画面に出るまでラグ
- 報道feed: daily cron（21:00 JST）、Geminiジャッジで関連度≥70・著作権≤30・誤情報≤30・重複≤50を通過したもののみ承認
- e-Stat: カタログ（表メタ＋リンク）のみ。数値はe-Govで誤読回避。鍵未設定なら非表示

## 2.3 通知性（新着が分かるか）— 最大の弱点

- **「前回訪問からの新着」表示なし。バッジ・未読数・メール/プッシュ・購読・RSS出力なし**
- 唯一の動的シグナルは報道feed（自動収集）。だが「何が新しくなったか」をユーザに知らせる仕組みは無い
- → 新着ハブ（doc05）＋RSS（doc06）＋「新着」バッジが最大の改善余地

## 2.4 トレンドAI要約の課題

- `computeAccidentTrend`は**curatedサンプル集計**で、月次速報を見ていない。「直近1か月」が0件になりがち
- オンデマンド（クリックで生成）・非キャッシュ
- ハルシネーションガードは強い（数値/固有名詞の創作禁止・少数注記・免責・no-key時はsummary:null）
- → 月次速報を要約の入力に加える／キャッシュ化が改善余地（doc07）

## 2.5 直近事故 公式リンク

- `official-recent-links.tsx`の**ハードコード5件**（月次速報・年次確定値プレス・死亡災害DB・死傷災害DB・e-Stat）。手動更新（コメントに2026-05-26確認）
- → `notice-url-health-check.mjs`（既存）でリンク死活監視に組み込む余地

## 2.6 「日本一使いやすい事故速報」への改善（→doc13）

1. **【P1】新着情報の一元化（新着ハブ）＋RSS** — 月次速報・報道feed・法改正(施行予定)・通達新着を1画面に。新着バッジ
2. **【P1】速報→トレンド連動** — 月次速報を`AccidentTrendSummary`の入力に。直近の実数で要約
3. **【P2】速報の反映ラグ短縮** — ETLコミット時にデプロイ誘発、または該当セクションを短revalidate ISRに
4. **【P2】e-Stat鍵の確認/明示** — 未設定なら意図的無効を明記、または設定
5. **【P3】直近公式リンクの死活監視** — 既存health checkに追加

## 2.7 結論

事故速報は**データ品質・ハルシネーション規律・網羅性が高い（手本級のETL）**。残る伸びしろは「速報性の見せ方（新着性・一元化・速報→トレンド連動）」であり、データではなく**提示と通知**の改善。いずれも既存データの再構成で実現でき、低リスク。
