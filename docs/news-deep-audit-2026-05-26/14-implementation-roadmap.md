# 14. 実装ロードマップ

## 14.1 段階

### Stage 1（本セッションPhase Bで着手）— 見せ方の核＋自動取得の核
- **P0-1 施行前/施行済 区別**（バッジ・フィルタ・カウントダウン）
- **P1-1 e-Gov法令API v2 自動取込ETL**（script＋Action＋マッパー、形式検証/未確認明記/diff-only）
- 検証: 新規テスト（ステータス判定・e-Gv v2マッパー）＋既存896維持

### Stage 2 — 一望性・通知（P1）
- P1-2 新着ハブ`/whats-new`＋ホーム訴求
- P1-3 RSS配信
- P1-4 事故速報→トレンド連動
- P1-5 法改正→チャットボット/化学物質RA動線

### Stage 3 — 仕上げ（P2）
- 影響業種拡充・タイムライン・反映ラグ短縮・AI要約安全統一・リンク死活監視・e-Stat鍵確認

### Stage 4 — 将来（P3・要判断）
- メールダイジェスト／多言語／官報・パブコメ誘導

## 14.2 各Stage完了条件（共通）

1. `npm run build`成功 / `npm run lint` 0 errors / `npx tsc --noEmit` 0 errors
2. `npm run test` 既存896維持＋新規pass
3. **法改正データは形式検証＋未確認明記＋推測値禁止を厳守**（取込は事実のみ、解釈はAI補助＋公式誘導）
4. branch→Draft PR→CI緑→squash merge→main pull
5. 本番デプロイ後に主要動線確認・既存6機能の非破壊確認

## 14.3 リスクと停止条件

- e-Gov API構造変化/取得失敗 → 既存手書き100件へフォールバック（pipelineは既にフォールバック設計）
- 誤読リスクが高い項目（内容解釈）→ 自動化せず公式リンク誘導に留める
- GitHub Actions: 月次cron＋diff-onlyで無料枠（2,000分/月）内
- Gemini: 件数限定＋キャッシュで月¥10,000内
- 新規env: e-Gov鍵不要のため**追加なし**（必要が生じたら社長判断）

## 14.4 ビルドコスト方針

- e-Gov ETLは月次速報同様「差分のみコミット・空ならコミットしない」
- RSS/新着ハブは既存データの静的生成（追加コスト極小）

## 14.5 順序の根拠

P0（施行前/済＝見せ方の致命的欠落）→ P1（自動取得＋一望性＝社長2大要件）→ P2（仕上げ）→ P3（投資判断）。「日本一使いやすく」は、**施行前カウントダウンで"気づかず違反"を防ぎ、新着ハブ/RSSで一望・通知、e-Gov自動取込で速報性・網羅性を底上げ**することで現実的に達成する。
