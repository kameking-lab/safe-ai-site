# 08. 多言語対応

## 8.1 重要な前提訂正

引き継ぎ指示は「サイネージ・化学物質RA・事故DBの6言語パターンを流用」を想定するが、**コード上、本サイトに6言語対応は存在しない**。

- `contexts/language-context.tsx`: `SUPPORTED_LANGUAGES = ["ja"]`。Providerは実質NOOPで`language:"ja"`固定、`setLanguage`はno-op、`t()`は常に`ja.json`
- i18nは**全機能でグローバル停止**（SEO-015/023コメント: EN切替は`/en/`SSR層が整うまで取り下げ）
- `ChatbotBody.tsx`のEN分岐・`language-button.tsx`・`translation-cache.ts`は**デッドコード**。想定言語もja/enの2言語のみで、6言語の痕跡はない

→ **「6言語パターン流用」は前提が成立しない**。チャットボット単独で多言語化すると、サイト全体ja固定との一貫性が崩れる。

## 8.2 チャットボットの現状

- パネル内の全文字列がハードコードJP。音声入出力も`ja-JP`固定
- リーフレットには`foreign-worker`ターゲット区分が存在（`leaflet-card.tsx:21`）＝外国人労働者向けニーズはデータ上想定済だが、UI言語は届いていない

## 8.3 多言語化の選択肢と評価

1. **何もしない（当面）**: サイト全体がja固定である以上、チャットだけ多言語化は不整合。**コスト制約・一貫性から当面はこれが妥当**
2. **質問入力の多言語受付＋ja回答**: Geminiは多言語入力を解釈可能。入力だけ多言語許容し回答はjaのまま、は小改修。ただし出典条文がjaのため実益限定
3. **回答の多言語化**: Gemini翻訳は追加コスト（多言語ぶんトークン増）。月¥10,000閾値・コスト管理（doc11）に直結。**社長判断事項**
4. **サイト全体の`/en/`SSR層**: 大規模。チャットボット監査の範囲外

## 8.4 結論・推奨

- **本Phaseでの多言語実装は非推奨**（サイト全体ja固定との不整合・コスト・前提誤り）
- 将来やるなら**サイト全体の言語基盤（`/en/`SSR層）と同時**に、まず外国人労働者の多い領域（化学物質・KY）から。優先度はP3
- 本Phaseでの関連作業は「`ChatbotBody`のデッドEN分岐の整理」程度に留める（→doc13 P3）
