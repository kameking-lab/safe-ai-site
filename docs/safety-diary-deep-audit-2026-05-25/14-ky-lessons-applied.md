# 14 KY機能の教訓を活かした設計指針

調査時刻: 2026-05-25

---

## 教訓1-7の適用（KYレビュー由来）

1. **環境制約を冒頭明記**：本docs群は全て冒頭で「実機未検証（ブラウザ無し・本番egress 403）」を明記済み。Web検索は機能＝競合は出典付き。**捏造YESなし**を貫いた。
2. **競合の架空化禁止**：03番は実在確認できたサービスのみ・出典URL＋調査時刻付き。料金等は「要確認」。
3. **「現場にめんどくさいをさせない」厳守**：打合せ書は元請が前日作成（元請負荷はOK・最小化）。**スマホ横スクロール入力を禁止**、AI事前チェックで「確認だけ」に。協力会社の当日実績返信は最小タップ。
4. **デッドコードを最初に特定**：`diary-detail-client`/`diary-print-client`/`updateEntry()` を検出（01番）。**入力経路の分裂**（/new と /new/detail で機能が割れている）も特定。
5. **2軸の判断**：11番（紙に戻る理由13件）・12番（他サービス理由10件）で全件抽出・重大度評価。
6. **P0/P1/P2/P3分類**：13番で工数・パターン付きで実施。
7. **確立パターンA-Iの流用**：下表。

## KY確立パターン A-I → 打合せ書への流用

| 記号 | KYで確立したパターン | 打合せ書での流用 |
|---|---|---|
| **A** | 用紙ファースト（WYSIWYG 1画面） | 北海道労働局様式を画面＝印刷として1画面で（P0-1） |
| **B** | ズーム（拡大縮小） | 広い各社マトリクスの全体↔部分（タブレット） |
| **C** | 自動取得（日付・天気・現場の履歴サジェスト） | ヘッダーの作業日/天気/現場/打合せ日を自動・履歴（P0-4） |
| **D** | プルダウン（共通選択肢） | 作業内容・使用機械・資格・災害をプルダウン＋履歴（P0-1,3） |
| **E** | マスター選択（作業員/業者） | **KYと共通マスター**で二重入力ゼロ（P1-8） |
| **F** | クラウド保管（Supabase＋localStorageフォールバック・device_id） | `meeting_records`＋オフライン耐性（P0-5） |
| **G** | 印刷/PDF（A4・確認印枠） | 官公庁様式A4・協力会社/作業所長の確認印枠（P0-6） |
| **H** | 6桁コード共有（signage_sessions） | 打合せ書を各社に配布・KY起点に（P1-7） |
| **I** | AI提案（Gemini RAG＋フォールバック＋grounded＋レート制限） | 災害/資格/リスク/指示/点検/法令の自動化（P1群） |

### KY最終仕上げ（直近実装）からの追加流用
- **過去一覧・再編集・複製（P0-A）** → 打合せ書の一覧/検索/複製/再編集（P0-6）
- **A4印刷＋確認印枠（P1-A）** → 打合せ書印刷（P0-6）
- **承認フロー・提出ロック（P1-B）** → 当日実績の提出ロック・元請確認（P1-6）
- **クロスツール連携クエリ（P1-C）** → 打合せ書↔KY↔事故DBの相互起票
- **同期状態表示（P1-D）** → 打合せ書のクラウド同期チップ

## 設計の一本筋
**「KYで作った部品をほぼ再利用して、書類だけ北海道労働局様式に差し替える」**のが最短・最安・最も品質が安定する。新規開発は「点検項目AIチェック」「指示文生成」「各社マトリクス」の3点に集中。
