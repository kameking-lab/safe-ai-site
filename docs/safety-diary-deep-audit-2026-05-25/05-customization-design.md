# 05 カスタマイズ機能（行追加・列追加・項目編集）の設計案

> 社長確定の"新発明"。元請ごとに自社書式へ微調整できるが、**他ユーザーに影響しない**保存が要件。実装はまだしない（本タスクは調査docsのみ）。設計妥当性のみ判定。

調査時刻: 2026-05-25

---

## 結論：実現可能。ただし「自由すぎるカスタム」は罠

- **行追加（業者・点検項目・有資格者）＝必須・低リスク**。すぐやるべき（KYの危険行追加と同型）。
- **項目の表示/非表示・ラベル変更＝中リスク・有用**。やるべき。
- **列追加（任意の新カラム）＝高リスク**。印刷レイアウト崩れ・AI非対応・PDF幅破綻を招く。**「定義済み列の表示切替＋自由メモ列1本」に絞るべき**で、無制限の列追加はやらない（弱腰でなく、明確に"やらない"判定）。

---

## データ構造案（フォーム定義をJSONで持つ）

書類本体（記入値）とは別に、**レイアウト定義（カスタマイズ）**を分離して持つ。

```ts
type FormCustomization = {
  formType: "ky" | "meeting"; // 打合せ書 = meeting
  version: number;
  // 定義済みフィールドの表示・順序・改名
  fields: { key: string; visible: boolean; label?: string; order: number }[];
  // 行を増やせるセクション（業者/点検/有資格者）の行数や追加行
  sections: { key: string; extraRows: number; customRows?: { label: string }[] }[];
  // 自由メモ列（1本まで）
  memoColumn?: { enabled: boolean; label: string };
};
```
- **記入値（打合せ書データ）は別テーブル/別キー**。カスタムを変えても過去データは壊れない（fieldsのkeyで突合）。

## 保存先：Supabase（KYと同じdevice_idモデル）

```sql
create table user_form_customizations (
  device_id text not null,
  form_type text not null,      -- 'meeting' | 'ky'
  payload   jsonb not null,     -- 上記 FormCustomization
  updated_at timestamptz default now(),
  primary key (device_id, form_type)
);
```
- **RLS**: 認証導入前は `device_id` スコープ（KYの `ky_records` と同方式）。**「ユーザー専用」は現状"端末専用"が正確**（本人認証は将来Phase）。この限界は社長に明記すること（捏造しない）。
- localStorageフォールバック（KYパターンF）も同梱。オフラインでもカスタム保持。

## UI設計案（KYの編集UXを流用）
- **行追加**: 各セクション末尾に「＋行を追加」ボタン（KYの危険行追加と同じ）。
- **項目編集**: 設定モード（歯車）で各フィールドに「表示/非表示」トグル＋ラベル鉛筆編集（インライン）。
- **並べ替え**: ドラッグは複雑＆スマホで誤操作。**上下ボタン**で十分（KY流の堅実UI）。
- **列**: 「定義済み列の表示切替」＋「自由メモ列ON/OFF」のみ。無制限列追加は不可（印刷保証のため）。
- **リセット**: 「公式版に戻す」ボタン（北海道労働局版がデフォルト）。

## 既存パターン流用
- KYの「マスター選択」「プルダウン」「自動保存」「クラウド＋localStorageフォールバック」をそのまま。
- 印刷（KYパターンG／最終仕上げP1-A）は**カスタム適用後のレイアウトで再現**できるよう、印刷用シートも `fields.visible/order` を読む実装にする。

---

## 強い判定
- **やる: 行追加・表示切替・ラベル改名・自由メモ列1本・公式版リセット**。これで「自社の打合せ書に寄せたい」9割の要望を満たす。
- **やらない: 無制限列追加・自由なセル結合・任意フォーム自作**（＝印刷とAIとPDFを壊す。Excel化への逆戻り）。
- 認証前は「端末専用カスタム」と正直に表示。多端末同一カスタムは6桁共有/将来認証で。
