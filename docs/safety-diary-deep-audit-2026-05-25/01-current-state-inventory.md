## 結論：既存 /safety-diary の正体

**＝「安全衛生日誌（職長日誌）型」のアプリ。社長確定要件の「安全工程打合せ書及び安全衛生指示書（北海道労働局版・元請現場代理人が前日に作る書類）」とは別物。**

根拠（データ構造 `web/src/lib/safety-diary/schema.ts`）:
- 必須5項目: 日付・天候・現場名・**作業内容**・**KY結果（/kyから転記）**・ヒヤリハット有無/内容
- 任意8項目: 業者別作業(名前と作業のみ)・必要資格・予定人数・予想災害・リスク評価(重大性×可能性の1つだけ)・安全指示事項・巡視記録・翌日予定

これは「その日の作業を1人（職長）が記録するログ」の構造。打合せ書が持つべき **各社マトリクス・使用機械・実績人員・実施状況・協力会社責任者サイン・作業主任者/有資格者一覧・8カテゴリ点検項目・使用機械リスト・作業所長/作成担当者/打合せ日** を **持っていない**。フィールド名が一部重なる（業者・予想災害・リスク評価）だけで、**書類の主体（元請 vs 職長）・タイミング（前日 vs 当日事後）・粒度（各社別 vs 全体1本）が根本的に違う**。

**現状完成度（"安全衛生日誌"として）: 55 / 100**（動くが、後述の致命的欠落あり）
**北海道労働局"打合せ書"としての達成度: 15 / 100**（構造の大半が未実装）

## 機能の全件リスト（コード確定）

### ルート（6本）
| ルート | 役割 | 状態 |
|---|---|---|
| `/safety-diary` (`page.tsx`) | 一覧＋当月統計（記録日数/延人数/KY実施率/ヒヤリ件数）＋JSON入出力 | **稼働** |
| `/safety-diary/new` | 必須5項目フォーム（3-5分・音声入力・AI下書き・ヒヤリ写真） | **稼働** |
| `/safety-diary/new/detail` | 必須5＋任意8の詳細フォーム（リスク5×5・業者行追加・資格自動推定） | **稼働** |
| `/safety-diary/[id]` | 個別詳細 | **死亡**（`permanentRedirect`→一覧。F-010で縮小） |
| `/safety-diary/[id]/print` | 個別印刷 | **死亡**（同上リダイレクト） |
| `/safety-diary/monthly/[ym]` | 月次サマリー（推移・上位災害・AI要約・印刷） | **稼働** |

### コンポーネント（6本）
- `diary-list-client` 稼働（一覧・統計・入出力）
- `diary-form-required` 稼働（必須・音声・`POST /api/ky-assist`・写真モーダル）
- `diary-form-detail` 稼働（詳細・`estimateQualifications()`・業者行追加）
- `diary-monthly-client` 稼働（月次・`window.print()`）
- `diary-detail-client` **デッドコード**（どこからもimportされない）
- `diary-print-client` **デッドコード**（同上）

### ライブラリ（lib/safety-diary）
- `schema.ts` Zod定義（稼働）
- `store.ts` localStorage `safety-diary-v3`（365件上限）。`updateEntry()` は **未使用（デッド）**
- `from-ky.ts` KY→日誌ローダー（稼働。`loadLatestKyForDiary()`）
- `presets.ts` 業種別サジェスト＋`estimateQualifications()`（正規表現ベース簡易推定・稼働）
- `monthly-summary.ts` 月次集計（稼働）

### 保存・連携
- **localStorageのみ。Supabase/クラウド保管なし**（KYは持っているのに日誌は無い＝大きな分断）
- 写真は別キー `safety-diary:photo:{id}`（1件のみ・データURL）
- KY連携は **双方向**: 日誌←KY（`?fromKy=latest`）/ KY←日誌（`?fromDiary={id}`、KY側 `deep-link-prefill.ts`）

## 致命的な欠落（"日誌"としても）

1. **保存した日誌を開けない・編集できない・1枚で印刷できない**（`[id]` と `[id]/print` がリダイレクトで殺されている）。一覧から個別を開く動線が無い＝**KYで直したばかりの「過去記録の再編集」と同じ後退がここにも在る**。
2. **クラウド保管ゼロ**。端末を変えると全消失。元請が事務所PCで作り現場でスマホ確認、が不可能。
3. **検索・絞り込みが無い**（一覧は新しい順固定）。現場名/期間で探せない。
4. **デッドコード3点**（`diary-detail-client` / `diary-print-client` / `updateEntry()`）が放置。

## 入力経路の分裂

- `/new`（必須）と `/new/detail`（詳細）の2フォームが**別実装**。どちらも同じschemaに保存するが、**詳細フォームに必須フォームの音声入力・AI下書き・写真機能が無い**（機能が割れている）。ユーザーは「軽い方」と「重い方」を選ぶが、重い方を選ぶと軽い方の便利機能を失う。**用紙ファースト1画面（KYのパターンA）に統合すべき。**

## KYとの依存関係マップ
- 日誌 → KY: `from-ky.ts` が `safe-ai:ky-instruction-record:v1`／`ky-record` を読む（読み取りのみ）
- KY → 日誌: `deep-link-prefill.ts`（KY最終仕上げで実装）が `getEntryById()` で日誌を読みKYに転記
- ダッシュボード `today-safety-dashboard` が当日日誌の有無を確認
- **結合は"文字列の流し込み"レベル**。共通の作業員/業者マスターは**未共有**（KYは `worker_master` を持つが日誌は使っていない）。
