# 06. タブレット実機検証 v2（軸6・768px相当）

**未検証の明示**: 見た目・タッチ感は実機未確認。以下はコード構造からの所見＋手順。

## コードから読める所見（768px）
1. **用紙幅820px ≈ 768px**：タブレット縦でほぼ原寸〜微スクロール。スマホより一望性が高く、**朝礼で複数人が覗き込む用途に向く**（zoomで拡大も可）。
2. **サイネージ（`/ky/morning`）はフルスクリーン・大フォント**（`text-3xl`〜`text-5xl`、リスクTop3・行動目標・指差呼称・唱和カウントダウン）。タブレットを立てて朝礼掲示に好適。
3. **別端末共有が刺さるデバイス**：事務所PC/職長スマホで作成 → **共用タブレットで `/ky/morning?code=XXXXXX`** を開けば、その端末にデータが無くても表示。前回「共用タブレットで何も映らない」を解消（実DB越しは未検証）。
4. **2カラム化**：用紙内グリッドは `sm:grid-cols-2`。768pxで2カラムが効き、横空間を活かせる。前回P2-6「2カラム発動が遅い」はこの用紙ビューでは `sm` で効くため概ね改善方向。

## 朝礼シナリオ（タブレット掲示）
1. 共用タブレットで `/ky/morning` を開く
2. 6桁コードを入力（職長が `/ky/paper` で発行したもの）
3. リスクTop3・行動目標・指差呼称が大表示 → 唱和カウントダウンで締め
- **複数人で見るのに適しているか**：フォント・コントラスト（白地に黒、リスクは赤帯）はコード上は視認性配慮あり。実機の輝度・距離での可読は未検証。

## 実機で必ず確認
- [ ] `/ky/morning?code=` が別端末タブレットで表示・8秒更新されるか
- [ ] 屋外/明るい現場での視認性（輝度・反射）
- [ ] 横向き/縦向き両対応の崩れ
- [ ] 用紙ビューの2カラムが768pxで破綻しないか

## 判定（軸6・暫定）
- **タブレットは本機能の"おいしい"デバイス**：原寸に近い用紙＋フルスクリーンサイネージ＋6桁共有が噛み合う。
- 実DB越し共有とサイネージ視認性の実機確認が取れれば、**「共用タブレット1台で朝礼が回る」**は競合（設置型クェスタ）に無料で対抗できる強み。
