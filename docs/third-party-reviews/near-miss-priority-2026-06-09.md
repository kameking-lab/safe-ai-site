# ヒヤリハット報告（/site-records/near-miss）第三者レビュー — 重大×未対策の優先可視化

実施日: 2026-06-09 / 軸2（既存機能を第三者目線で使いやすく）

## ペルソナ
**畑山さん（54歳・ゼネコン作業所の専任安全衛生責任者）**
- ハインリッヒの法則を信じ、「重大災害は無数のヒヤリの上に立つ」が口癖。ヒヤリ報告を毎日集める。
- 一番恐れているのは「重大の可能性があるヒヤリを集めたのに、対策しないまま放置して本災害になる」こと。過去に他現場で開口部の転落ヒヤリを潰しそびれた事例を見ている。
- 我慢の限界: 「報告は溜まるのに、どれを今すぐ手当てすべきかが一目で分からないなら、紙の付箋を壁に貼った方がマシ」。スマホ（iPhone・390px）で朝に確認する。

## 操作（Playwright・iPhone 390×740 実機相当）
畑山さんが1週間で集めた5件を投入（実アプリの保存と同じ日付降順で保持）:
- 05-28 墜落・転落／3F開口部（重大・**未対策**）
- 06-01 はさまれ・巻き込まれ／資材ヤード（重大・**未対策**）
- 06-06 飛来・落下／足場下（重大・**未対策**）
- 06-07 切れ・こすれ／加工場（軽微・対策済）
- 06-08 転倒／通路（軽微・対策済）

## 指摘（忖度なし）
1. **【致命】対策すべきものが最下段に沈む。** 一覧は日付降順で固定のため、表示順は
   `1.転倒(軽微・対策済) → 2.切れ(軽微・対策済) → 3.飛来(重大・未対策) → 4.はさまれ(重大・未対策) → 5.墜落(重大・未対策)`。
   畑山さんが一番手当てしたい **墜落・転落（重大・未対策）が5件中5番目＝最下段**。すでに片付いた軽微案件が画面の一番上を占有する。報告が20件溜まればスクロールして探すことになり、重大ヒヤリが視界から消える。
2. 「対応中 3件」バッジは件数を示すだけで、その3件が**どこにあるか・どれが重大か**へ導かない。
3. 重大・未対策のカードに視覚的な強調がなく、軽微・対策済と同じ並びの中で埋もれる。
4. 未対策だけを抜き出して確認する手段がない（対策済を一件ずつ読み飛ばす必要）。

→ 畑山さん初回評価: 「集計の棒グラフはいい。でも一覧が日付順だけなら、結局どれを先に潰すか自分で全部読み直す。これなら危険なヒヤリだけ壁に貼る」。

## 改善（実装）
- 純関数を `nearmiss-store.ts` に追加（テスト8件、ロジック/データ/スキーマ不変）:
  - `priorityRank(r)` … 未対策×重大=0 / 未対策×軽微=1 / 対策済×重大=2 / 対策済×軽微=3
  - `sortByPriority(reports)` … ランク昇順→同ランクは日付降順→savedAt降順（元配列を破壊しない）
  - `openHighCount(reports)` … 重大×未対策の件数
  - `filterOpenOnly(reports, openOnly)` … 未対策のみ抽出
- 一覧セクションを改修:
  - 表示順を `sortByPriority` に変更＝**重大×未対策を必ず先頭**に集約。
  - 重大×未対策が1件以上あるとき、一覧先頭に赤い**要対策アラート**（「重大の可能性×未対策が N 件あります。優先して対策してください」）を表示。
  - 重大×未対策カードに**赤い左ボーダー＋「要対策」赤バッジ**を付与し、対策済（緑）・軽微と一目で区別。
  - **「未対策のみ表示」トグル**を追加＝対策済を畳んで未対策だけに集中できる。
- 集計・CSV・印刷・保存スキーマ・入力フォームは不変。新規依存0。

## 再レビュー（Playwright実機）
- 一覧の並びが `1.飛来(要対策) → 2.はさまれ(要対策) → 3.墜落(要対策) → 4.転倒(対策済) → 5.切れ(対策済)` に変化。**重大×未対策3件が先頭に集約**、対策済が末尾へ。
- 先頭に「重大の可能性×未対策が3件あります…」の赤アラートを表示。各要対策カードに赤左ボーダー＋「要対策」バッジ。
- 「未対策のみ表示」ON→未対策3件のみ（飛来・はさまれ・墜落）に絞り込み。
- 横はみ出し 0px（iPhone 390）。
