実施日: 2026-06-08 / ブランチ: review/accidents-consultant-quicksearch

## ペルソナ
**佐藤(58)・独立系の労働安全コンサルタント。** 顧問先10社を掛け持ちし、移動中のスマホで「明日の現場説明用に、ある事故型の実例をすぐ出したい」。
- 性格: せっかち。10秒で目的に辿り着けないツールは「使えない」と判断し二度と開かない。
- 前提: 「職場のあんぜんサイト」を長年使ってきたが、PC前提・検索が遅いのが不満。スマホで完結したい。
- 我慢の限界: **目的の検索窓に辿り着くまでにスクロールを何度もさせられること。** 「全件検索」と書いてあるのに検索窓が無いこと。

## 実操作（Playwright・iPhone相当 390×844／dev server）
1. `/accidents` に着地 → ページ全高 **10,238px**。
2. 最初の画面〜数スクロール: 表示モード説明 → タイトル → 出典内訳の長い定義文 → AI分析 → 直近公式リンク → トレンド → 保存 → はしご統計 → ニュースフィード…と続き、**検索/絞り込みのタブ列に着くのは先頭から 5,561px 下（モバイルで約6.5画面）。**
3. しかも既定タブのラベルは **「全件検索 (504,415件)」** なのに、その中身は「自社類似事故Top5(要プロファイル設定)」＋「業種×事故型クロス集計」で、**キーワード入力欄が存在しない(検証: `#accident-keyword` = なし)。**
4. 実際にキーワードで事例を引くには、さらに別タブ「サイト収録事例(292件)」を探して開く必要がある。

## 忖度なしの指摘
- **(致命)** 検索が深すぎる。「事例をすぐ出したい」コンサルが、検索窓に着くまで6画面以上スクロールさせられる。佐藤は確実に離脱する。
- **(致命)** ラベル詐欺。「全件検索」タブに検索窓が無い。期待を裏切る。
- (中) 検索窓は「サイト収録事例」タブの中に隠れていて、名前から検索機能だと分からない。
- (小) 先頭の出典内訳の定義文が長く、ファーストビューを圧迫している。

## 改善実装（本PR）
最小・非破壊で「致命」2点に対処。**既存破壊なし**（既存タブ・URL同期ロジックはそのまま再利用）。
1. **最上部に「🔎 事例をすぐ検索」カードを新設**（`components/accidents/quick-accident-search.tsx`）。
   - キーワード入力＋「検索」ボタン。
   - 頻出事故型の1タップチップ（墜落・転落／転倒／はさまれ／飛来・落下／熱中症／感電）。
   - 遷移先は `/accidents?tab=list&acc_kw=...#accident-results`。**tab=list・acc_kw は AccidentDatabasePanel が既にマウント時復元する実装済み導線**を再利用（フル遷移で HomeScreen 再マウント→タブ選択＋キーワード復元＋結果へスクロール）。
2. **結果パネルにアンカー `#accident-results` を付与**（`accident-database-panel.tsx`）。区画先頭(#section-accidents)だと自社Top5・クロス集計の下に結果が埋もれるため、キーワード欄＋絞り込み結果の先頭へ直接着地させる。

## 再レビュー（改善後・同条件で実操作）
- クイック検索カードの位置: **5,561px → 1,080px**（ファーストビュー直下、約1スワイプ）。
- 1タップ「墜落・転落」→ URL `?tab=list&acc_kw=墜落` に遷移、**キーワード欄に「墜落」が入った状態で 42件 の絞り込み結果へ自動スクロール着地**（scrollY≈7283＝結果パネル先頭）。業種・属性・規模・作業カテゴリ・事故種別の絞り込みも同画面に揃う。
- 佐藤の評価: 「着地して1タップで目的の実例が並ぶ。これなら移動中でも使える。」→ **採用ラインに到達**。

## 残課題（次サイクル候補・本PR範囲外）
- 既定タブ「全件検索」のラベルは依然ミスリード（中身はTop5＋クロス集計）。クイック検索で主導線は確保したため致命度は低下したが、ラベルの正直化（例:「傾向・自社向け」）は別タスクで検討。
- 先頭の出典定義文の畳み込み（detailsで初期折りたたみ）でファーストビューをさらに軽くできる。
