# 保護具AIファインダー(/equipment-finder) 第三者レビュー — 選定根拠の分かりやすさ

- 日付: 2026-06-09
- 対象: `/equipment-finder`（保護具AIファインダー）
- 観点: 絞り込みの速さ・選定根拠の分かりやすさ
- 手法: 厳しいペルソナになりきって Playwright で実機操作（iPhone 12 / 390×844、結果カードは 768幅でも確認）

## ペルソナ

**田中さん（48歳・中堅ゼネコンの専任安全衛生管理者）**

- 足場・鉄骨の高所作業に向けたフルハーネスを至急選定し、社内の「保護具選定記録」に根拠を1行で残したい。
- 一番嫌うのは「結局その製品が法令・規格に適合しているのか、画面で確認できない」こと。製品リンクを1つずつ開いて検定マークを探すのは現場では時間の無駄。
- ファインダーのトップに「JIS規格・国家検定品も明示」と書いてあるのを見て期待して使う。

## 実機操作と指摘（忖度なし）

Playwright で実際に `種類選択(フルハーネス) → 絞り込み(X型/シングル/建設) → おすすめ商品` を操作。

### 致命的: 「選定根拠」の核心＝法令・規格の根拠が結果カードに一切出ない

- ヘッダー／メタは「**JIS規格・国家検定品も明示**」と明言しているのに、結果カードに表示されるのは
  `メーカー名 / 製品名 / subCategory / spec / マッチpt / 価格 / 評価 / おすすめ理由 / マッチした回答` だけ。
- DB の全 1,050 件が持つ `regulations`（根拠法令・告示。例: 「安衛則 第518条」「墜落制止用器具の規格」「石綿則 第14条」）が**どこにも描画されていない**。
  実機の結果ページ本文を grep しても `安衛則 / 告示 / 規格` がヒットせず＝約束が果たされていない。
- さらに `spec` は製品によって「JIS T 8165:2018 第一種」のように規格が入る一方、「墜落制止距離 1.85m以下」のように**規格でない性能値**が入る個体もあり、検定/規格の有無を spec だけでは判断できない。
- 結果: 安全担当が一番欲しい「この製品は法令・規格に適合しているか／何法が根拠か」を、画面では確認できず製品リンクを1件ずつ開くしかない。「選定根拠の分かりやすさ」が成立していない。

### （参考・今回は範囲外）フルハーネス種別にランヤードが混在

- 「フルハーネス」カテゴリは `subCategoryIncludes: ["フルハーネス","ランヤード"]` で**ハーネスとランヤードを同居**させている。
- ハーネス専用属性「X型」を選んでも、ランヤード（形状の概念なし）が `シングル+建設+高評価=70pt` でハーネスと同点になり、レビュー数で上位に出る＝**X型ハーネスを探したのにランヤードが1位**。
- これはスコアリング（推薦妥当性）の問題で副作用が広いため、今回は是正せず BACKLOG に切り出し。

## 改善（インパクト大の1点に集中）

結果カードに「**適合区分バッジ**」と「**根拠法令・規格**」を追加し、ヘッダーの約束（JIS規格・国家検定品も明示）を実際に果たす。

- 純関数 `src/lib/equipment-finder/compliance.ts` を新設（テスト10件）。
  - `deriveComplianceBadge(spec, regulations)`: DB に**実在するマーカー文字列のみ**を根拠に1バッジを導出。
    優先度 `国家検定品 > 労検合格品 > 型式承認品 > JIS規格適合`、該当無しは `null`（**捏造しない**）。
    データ実測カバレッジ: 国家検定115・労検47・桜マーク/型式16・JIS580＝**758/1,050件**にバッジ、残りは非表示。
  - `formatRegulations(regulations, limit=4)`: trim・空除去・重複除去して根拠チップ用に整形。
- 結果カード(`equipment-finder-client.tsx`)に
  - spec の隣に適合区分バッジ（国家検定/労検/型式承認=amber強調、JIS=blue）。
  - おすすめ理由の下に `⚖ 根拠法令・規格` ボックスを新設し regulations をチップ表示。
- 既存の `recommendItems`・スコアリング・データ・スキーマ・アフィリエイト導線は**一切不変**（純追加表示）。新規依存0。
