状態性が薄い4画面（手順書・死傷病報告・資格管理簿・カレンダー）を、それぞれに効く文法で結論カード化した。

## 画面別の設計判断（タスク指示「効果のある形で」への回答）

| 画面 | 採用した文法 | 判定関数（純関数・テスト固定） |
|------|--------------|------|
| 手順書 /site-records/procedure | KYと同じ「記入のこりN項目（青）→ 記入完了（緑）」。**全欄空の手順行は未使用として数えない**（既定3行が常に9項目残りになる偽カウント防止）。書きかけ行の空欄＋作業名＋「手順が1行も無い」を数える | `countProcedureRemaining` / `procedureConclusion` |
| 死傷病報告 /site-records/incident-report | 「記入のこりN欄（青）→ 下書き完了（緑）」。13欄固定（備考は任意で数えない）。**完了＝提出ではない**ため、緑の補足に「提出は電子申請」を必ず明記（2025年1月から原則義務化） | `countIncidentRemaining` / `incidentConclusion` |
| 資格管理簿 /site-records/qualifications | 期限概念が無くデカ数字の種が無い→「登録N名・資格M種」の現況カード（青）＋**逆引き名簿への44px動線**（#qual-lookupアンカー新設）。登録ゼロは「登録なし」の案内 | `qualificationsConclusion` |
| カレンダー /site-records/calendar | 「今月のこりN件（青）→ 今月完了（緑）」。今月の定例項目に**消し込みチェックを新設**（localStorage・月替わりで自動リセット・直近12か月だけ保持）。リンクは「開く」として別タップ対象に分離 | `calendarConclusion` ＋ calendar-progress.ts |

色の文法: この4画面には停止級（赤）・要対応（黄）の状態が存在しない（手順書の空欄は危険ではなく作業前の準備）。
青＝案内・緑＝完了のみで構成し、赤黄の価値を薄めない（第1弾のヒヤリ「報告ゼロは緑にしない」と同じ思想の裏返し）。

## 無読テスト（社長指示 2026-06-10・全レビュー必須）

スクリプト: `docs/third-party-reviews/scripts/records-noread-phase2-2026-06-11.mjs`（実UI操作でデータ作成・localStorage直書きなし）
実行: 本番ビルド（npm run build → npm run start）に対して 2026-06-11 実施

**結果: 21/21 PASS**

- ①ファーストビュー: 4画面とも結論カードが390×844の初期表示内（y=416〜495）
- ②色の即時反映: 手順書=作業名記入で2→1項目に即減→1行記入で緑/死傷病=12欄で「のこり1欄」→13欄で緑/資格=保存で「登録1名・1種」/カレンダー=全消し込みで緑→**リロード後も緑（永続化）**→1件解除で青に復帰
- ③タップ対象: 資格「資格から逆引き」・カレンダー「今月の項目へ」とも44px、デカ数字48px

## 申し送り（今回は対象外・将来の改善候補）

1. 資格管理簿に取得日からの**経過年数表示**（技能講習は失効しないが、再教育の目安〔おおむね5年・通達ベース〕の可視化は価値がある。法令正確性の確認込みで別タスク）
2. カレンダーの消し込みは端末ローカル。複数端末共有はKY同期と同じPath A制約（オーナー判断待ちのSupabase拡張に相乗りする形が自然）
3. 死傷病報告の結論カードは様式23/24号の別を表示していない（カード上は同じ13欄。様式区分はヘッダーのセレクトで可視）

## 不可侵の確認

- A4印刷帳票: 結論カードは全画面 `print:hidden`。手順書・死傷病・受入教育の帳票出力は不変
- 法令記述: 既存文言に手を加えていない（死傷病の電子申請義務化の記述は既存の注意書きと同一内容をカード補足に転記）
- 既存破壊なし: 保存スキーマ・CSV出力・印刷フロー不変。新規localStorageキーは `safe-ai:calendar-done:v1` のみ追加
