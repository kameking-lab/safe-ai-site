# 第三者レビュー: サイネージ全画面 /signage/display を「無人で1日流しっぱなし」前提で点検

- 日付: 2026-06-09
- 対象: `/signage/display?fullscreen=true`（`SignageMapClient` フルスクリーン表示）
- 軸: 軸2（既存機能を第三者目線で使いやすく）

## ペルソナ
**佐藤さん（48歳・地場ゼネコンの安全担当）**
- 現場事務所のタブレットに気象警報・地震マップを映し、**朝置いたら夕方まで誰も触らない**運用をしたい。
- ITは詳しくない。「気づいたら画面が真っ暗」「映ってるけど実は朝の情報のまま」が一番困る。
- 我慢の限界: ①昼に見て画面が消灯していたら「使えない」と判断し電源を切る。②表示の説明文と実挙動が食い違うと不信感。③回線が一瞬切れたまま夕方まで古い情報を映し続けるのは事故の元。

## Playwright 実機操作（1920×1080・フルスクリーン）で見つけた欠点
1. **【致命】画面スリープ抑止が一切働いていない。** `navigator.wakeLock.request` の呼び出し回数を実測 = **0**。プロジェクトには常設サイネージ用の `useWakeLock`（`src/lib/signage/use-wake-lock.ts`）が存在し朝礼サイネージ(`ky-morning-signage`)では使われているのに、**肝心の無人フルスクリーン表示 `/signage/display` では未結線**。タブレットのOS消灯タイマー（通常5〜10分）で暗転し、「無人で1日流しっぱなし」が成立しない。
2. **【要改善】自動更新の説明が実挙動と食い違う。** フッターに「**15分**ごとに自動更新」と表示されるが、実際の `REFRESH_INTERVAL_MS` は **30分**（コード上も30分が意図的と明記）。佐藤さんが「15分前の情報のはず」と誤認する。
3. **【要改善】取得失敗時の回復が最大30分の盲点。** 回線断（`/api/signage/jma` を abort で再現）で「取得エラー」バッジは出るものの、次の定期更新まで**最大30分**古い情報を映し続ける。無人運用では復旧の取りこぼしが痛い。

## 改善実装（既存資産の再利用・新規部品0）
- **①画面スリープ抑止を結線**: `SignageMapClient` に `useWakeLock(initialFullscreen)` を追加。無人フルスクリーン表示(`/signage/display`=既定 fullscreen)でのみ作動し、地図編集モード(`/signage/map`=`initialFullscreen={false}`)では抑止しない。非対応ブラウザでは無害な no-op（既存フックの設計どおり）。
- **②説明文を実挙動に一致**: 「15分ごとに自動更新」→「**30分ごとに自動更新（タブが表示中のときのみ。取得失敗時は3分後に再試行）**」。
- **③取得失敗時の短間隔リトライ**: 取得エラー時のみ `ERROR_RETRY_INTERVAL_MS = 3分` 後に1回再試行をスケジュール（タブ非表示時はスキップ、アンマウントで確実に clearTimeout）。成功すれば30分の盲点が3分に縮む。定期更新ロジック自体は不変。

いずれも**気象/地震データの取得・描画ロジックは不変**。1画面フィット（フルスクリーン `h-screen w-screen`）も変更なし。

## 再レビュー（Playwright 実機）
- `wakeLockRequested = 1` ＝ **画面スリープ抑止が作動**（佐藤さん「昼に見ても消えてない」）。
- フッター = 「30分ごとに自動更新（タブが表示中のときのみ。取得失敗時は3分後に再試行）。」＝**実挙動と一致**。
- 回線断シナリオ（`/api/signage/jma` abort）で「**取得エラー**」バッジ表示を確認＝異常が一目で分かる。失敗時は3分後に自動再試行（コード経路で確認）。

佐藤さん「これなら朝置いて夕方まで任せられる。画面も消えないし、回線が戻れば勝手に直る」＝採用ライン到達。
