# 04 スマホ実機検証（375px相当）（軸4）

検証手段: Playwright(Chromium) で本番 https://www.anzen-ai-portal.jp/ky/morning を 375×812 で実レンダリング（ky-record をlocalStorageに注入し朝礼コンテンツを表示）。スクショ: `screenshots/smartphone-375.png`。

注: Lighthouse CLI / Chrome バイナリが当環境に無く、Lighthouse スコアは**未測定**（捏造しない）。本番デプロイ環境での Lighthouse 実行を別途推奨。

## 実測値（系統1 /ky/morning, 375px）
- 横スクロール: **なし**（docWidth=375=innerWidth）✓ レイアウト崩れゼロ
- コンテンツ幅: 375px（フル）
- 「本日の主な作業」見出し: 30px（base `text-3xl`）— スマホでは妥当
- リスクTop3カード: 縦積み、border-l-8 で視認性良好
- 唱和カウントダウン: `text-7xl`（72px相当）で大きく見やすい

## 良い点
- mobile-first で破綻なし。タップターゲット（開始ボタン等）は大きい。
- 6桁コード入力 inputMode=numeric / pattern で数字キーボード誘発 ✓
- 暗背景(slate-900)+白文字＋カード白背景でコントラスト良好。

## 課題（スマホ）
- S-1: スマホで朝礼サイネージを「全画面投影」する場合、ブラウザのアドレスバー/ステータスバーが残る。PWA/フルスクリーンAPI（`requestFullscreen`）導線なし。
- S-2: 画面スリープ抑止（Wake Lock API）なし → 朝礼中に画面が暗転しうる。専用サイネージ機に対する明確な弱点（軸2）。
- S-3: 唱和ビープは1回のみ。スマホスピーカーは小音量で朝礼の屋外現場では聞こえにくい。
- S-4: 縦持ち前提。横持ち(landscape)時のレイアウト最適化指定なし。

## 系統2 /signage のスマホ
- 「この画面はPC・大画面TV表示用です」バナーで期待値調整済み。縦スクロールで全セクション確認可。
- ただしスマホ単体で朝礼投影には不向き（情報密度が高くダッシュボード志向）。

## スマホ総評
系統1はスマホ朝礼表示として**実用レベル**。専用機に対する弱点は「フルスクリーン化」「画面スリープ抑止」「音量」。S-1/S-2 はWeb標準APIで安価に解消可能（Phase C候補）。
