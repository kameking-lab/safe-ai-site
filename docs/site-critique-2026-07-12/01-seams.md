# 01 統合の縫い目 — 「機能の足し算」は線として繋がっているか（本番実測）

- 対象: https://www.anzen-ai-portal.jp/ （2026-07-12 実測）+ リポジトリ /home/user/safe-ai-site
- 手法: curl による全HTML/リンク抽出 + Playwright（モバイル390px・一部PC1920px）実操作。スクリーンショット20枚はレビュー時に取得・確認済み（容量のためリポジトリ非収載）。
- 重大度: **S**=体験が破綻 / **A**=迷子になる / **B**=惜しい。規模感: S/M/L（修正工数）。
- **重大度サマリ: S=2・A=5・B=3。**

総括: 各機能は単体では出来ているが、**「発見の層」（グローバルナビ・機能一覧・横断検索・チャットボット回答）が並列開発に追随しておらず、新しい機能ほど孤島化している**。法令ナビ／現場ことば版という本サイトの差別化機能が、初見ユーザーの通常動線からは実質的に存在しない。

## 縫い目1: /law-navi・/law-search・/search の三つ巴 【S】

### 実測した事実
- **/law-navi はグローバルナビに存在しない。** トップページHTML内のリンク143本に `law-navi` は0件。モバイルメニュー（スクショ03）にも、フッターにも、機能一覧 `/features`（「全機能」を謳うカタログ）にも0件。到達路は (a) /laws・/law-search・/glossary・/circulars 上部の「法令まわり」ピル（`components/law-hub-nav.tsx`）、(b) 横断検索の「機能」ヒット、の2本のみ。
- **/search（横断検索）はアイコンのみ。** モバイルはヘッダーの虫眼鏡アイコン（aria-label「検索を開く（Ctrl+K）」）、PCは右上「検索 Ctrl+K」。テキストリンクは home/features/menu/footer すべて0件。/law-navi の検索窓が `action="/search"` のGETフォームで合流しているのが唯一のページ内導線。
- **/law-search だけが正規メンバー。** メニュー「学ぶ>法令検索」・トップ・フッターに露出。
- 役割説明は law-hub-nav ピルの `title` 属性（ツールチップ）にあるが、**モバイルではタップでツールチップが出ないため実質不可視**。3つを並べて使い分けを説明する場所はどこにもない。
- 同一クエリ実測:
  - 「フルハーネス」: /search=49件（法令3・法改正9・FAQ5・通達4・保護具22・機能1…先頭に law-navi topic）／ /law-search=3条文（36条・518条・521条）／ /law-navi=fall-arrest 分野タイル（条文9+通達10）。
  - 「石綿 事前調査」: /search=16件 ／ /law-search=1件（石綿則3条）。
- **/search の「法改正」ヒットは全部 `/laws`（一覧ハブ）に落ちる**（`lib/search-index.ts` L213 に「個別詳細ページ未実装のため /laws へ寄せる」と明記）。9件の法改正ヒットのどれをタップしても同じ一覧の先頭に着地し、見ていた項目を自力で探し直すことになる。

### 初見ユーザーの実害
「法令を調べたい」人がメニューから届くのは /law-search（1,076条文の全文検索）だけ。フルハーネスで3件しか出ず「このサイトは条文が薄い」と誤認して離脱する。実際には49件出る横断検索と、体系順に読める法令ナビが裏に居るのに、その存在を知る手段がない。3機能を全部知っている人（=作った本人）にしか成立しない導線。

### 直し方
1. `components/app-shell.tsx` のナビ定義「学ぶ」グループに `/law-navi`（法令ナビ — 現場の言葉から条文へ）を追加。`/features` のカタログデータにも追加。
2. /law-search の検索結果が少ない時（例: 5件未満）に「横断検索で49件ヒットしています → /search?q=」のインラインバナーを出す（逆も同様に /search の法令タブから law-search へ）。
3. 三者の使い分けを law-hub-nav の title 属性から**可視のテキスト**（1行キャプション）に昇格。「ナビ=分野から読む / 条文検索=全文 / 横断=サイト全部」。
4. /search の法改正ヒットに `/laws#<改正id>` アンカー（一覧側に id 付与）で着地精度を上げる。詳細ページ実装を待たない応急処置として S。
- 規模感: **S**（ナビ・カタログ追加とバナーはデータ追加レベル。4はアンカー付与のみ）

## 縫い目2: 化学ワンボックス /chemical-ra と旧 /chemical-database 【B】（ゼロヒット時のみ A）

### 実測した事実
- トルエン: /chemical-ra?name=トルエン&run=1 → 有機則第二種・毒劇法劇物・PRTR第一種・RA対象を根拠条文付きで即答。ページ冒頭に「このページの使い分け: 詳細閲覧だけなら化学物質検索DBへ」の案内あり。DB詳細（/chemical-database/108-88-3）には「RAを開始」（cas引き継ぎ付き `/chemical-ra?cas=108-88-3`）あり。双方向は繋がっている。
- **カプサイシン（収載外物質）で挙動が割れる:**
  - /chemical-ra → 「収載外です」+ **AI詳細調査 / SDS読み取り / 製品名から成分検索** の3つの脱出路 + 「収載外でも安全とは限らない（57条の3・28条の2）」の注意。
  - /chemical-database → 「**該当する物質が見つかりませんでした。別の語句でお試しください。**」で終わり。直下に表示されるのはアフィリエイトの防塵マスク商品カード（スクショ14）。RA側のAI調査への誘導なし。
- /chemical-database は検索語をURLに持てない（クライアント側フィルタのみ、`?q=` 非対応）。他機能から「この物質でDBを開く」深リンクが物質詳細ページ経由でしか組めない。

### 初見ユーザーの実害
未知の物質を調べた人が、入口をDB側にした場合だけ「無い」で止まり、義務の可能性（RA努力義務/義務）も次の一手も知らされない。同じサイトの隣のページには救済動線が実装済みなのに、入口の選び方で結果が変わる。ゼロヒット直後の商品広告は「調べ物に失敗したら物を売られた」体験になる。

### 直し方
1. `components/chemical-database-client.tsx` のゼロヒット表示に、chemical-ra と同じ3ボタン（AI詳細調査/SDS読み取り/製品名検索）+ 57条の3の注意文を移植（コンポーネント化して共有）。アフィリエイト枠はゼロヒット時は非表示に。
2. 同コンポーネントに `useSearchParams` で `?q=` 初期値対応を足し、RA・横断検索から深リンク可能にする。
- 規模感: **S〜M**

## 縫い目3: 現場ことば版（plain）の発見性 【S — 孤島と断定】

### 実測した事実
- 現場ことば版は `/law-navi/[lawId]/[artSlug]` 条文ページ内のセクションとしてのみ存在（実測: /law-navi/347AC0000000057/59 に表示、スクショ10）。データは `web/src/data/plain/` に19法令分整備済み。
- **「現場ことば」という語はサイトの他のどこにも出ない**: home/features/whats-new で0件。
- 横断検索 `/search` の条文ヒットは `/law-search?law=..&art=..` に着地する（`lib/search-index.ts` L179）— **現場ことば版がある law-navi 条文ページには行かない**。plain本文は検索インデックスに未収載。
- /law-search の結果カードのリンク先は e-Gov と自分自身の絞り込みのみ（`components/law-search-results.tsx`）— law-navi 条文ページへのリンクなし。
- チャットボットの出典リンクも全て `/law-search?q=`（`lib/chatbot-enrichment.ts` L184-310）— plain へ0本。
- 外部からの実質的な入口は「law-navi 分野→条文」と、教育スライド詳細の根拠条文リンク（/education/hazard-slides/fall → /law-navi/347M50002000032/518 等）のみ。そして縫い目1の通り law-navi 自体が孤島。**孤島の中の孤島。**

### 初見ユーザーの実害
「条文が読めない現場の人向け」がこの機能の存在理由なのに、その人が通る道（横断検索・条文検索・チャットボット）のどこからも見えない。19法令分の翻訳資産がほぼ回収されていない。

### 直し方
1. `lib/search-index.ts` に plain 条文をソース追加（title=「◯◯法 第◯条（現場ことば）」、url=law-navi条文ページ）。data/plain は純データなので dynamic import 1本で載る。
2. `components/law-search-results.tsx` の結果カードに、plain が存在する条文（`getFreshPlainArticle` で判定可）だけ「現場ことば版で読む →」バッジを追加。
3. チャットボット回答の参照条文カード（`chatbot-panel.tsx` の citations）に同判定で law-navi 条文ページへのリンクを併記。
- 規模感: **M**（3ファイル、判定関数は既存）

## 縫い目5: 通知ベル 【A】
