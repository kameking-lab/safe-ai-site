## 0. 要約（結論先出し）

- **課題**: 横断検索/`/law-search` が引く curated コーパスは **抄録 1,065 条**。安衛則は実際には第677条まで存在するのに未収録条番号があり（実測: `第61条の2`・`第577条の2` → 「該当なし」）、**「法令上存在しない」と「当サイト未収録」を利用者が区別できない**＝「規定がない」と誤読させる安全上のリスク（診断 G7）。
- **短期（実装済み・S6）**: 0件時に e-Gov フォールバック＋収録範囲の明示で **誤読を止血** 済み。これは信頼の防波堤であって、カバレッジ穴そのものは残る。
- **中期（本ドラフトの対象・T7）**: e-Gov 法令API v2（**認証不要・XML全文・政府標準利用規約2.0＝商用可/出典明示**）から中核法令の**全条文**をビルド時に取込み、抄録→全文へ引き上げる。速さ（実測10〜30倍）×完全性で e-Gov の最後の優位（カバレッジ）を消す。
- **本ドラフトの結論**: 技術的には既存インフラの再利用でほぼ賄える（後述の再利用資産§3）。**新規 env は不要**（API キー無し）。ただし (a) 外部API本番取込みの是非、(b) 全文取込みによるクライアントバンドル肥大＝LCP/PWA プリキャッシュへの影響、(c) データファイルの所有（`web/src/data/laws/**` は data 班領域）、の3点は**オーナー/他班判断**。SEO班単独で実装着手はしない。

## 2. ゴールと完了条件（診断 T7 の受入基準を具体化）

1. **カバレッジ**: 安衛則の任意の実在条番号（例 `第577条の2`）が検索でヒットし、条文本文が返る。「未収録による該当なし」を中核法令から解消。
2. **原文一致**: 取り込んだ本文が e-Gov 原文とサンプル照合で一致（改ざん・要約・欠落0）。既存の `egov-caption-snapshot.ts`＋`article-caption-integrity.test.ts` と同じ「公式スナップショット照合」方式を本文へ拡張。
3. **速度非退行**: `/law-search` の応答が現行水準（<200ms）を維持。全文化でバンドルが肥大するため、**検索ワーカー化 or 遅延ロード**を併設（§5）。LCP 悪化なし。
4. **出典・ライセンス**: 各条に e-Gov URL と取得日を付与。**政府標準利用規約2.0（商用可・出典明示）**を明記。政府公開ソースのみ（loop-prompt ライセンス厳守）。
5. **法令正確性の不可侵**: 形式データのみ機械取込み。解釈・要約は行わない。取得失敗条は**スキップして欠落を明示**（捏造・推測日の創作をしない＝`egov-revisions-fetch.ts` の設計哲学を踏襲）。

## 4. 取込みパイプライン設計（実装時の青写真）

### 4.1 方式: ビルド時取込みではなく「オフライン生成＋チェックイン」

診断は「ビルド時生成」と表現するが、`scripts/law-data-import/README.md` が既に規定する通り **Vercel ビルドは外部ネットワーク到達を前提にしない**（ビルドの再現性・CI 安定性・main 常時デプロイ可能の担保）。したがって:

- **取得スクリプト**（新規 `scripts/etl/egov-fulltext-fetch.ts`、`egov-revisions-fetch.ts` と同じ作法）を**手動 or Cron で実行** → `web/src/data/laws/<law>.generated.ts`（または JSON）へ書き出し → **git にチェックイン**。
- ビルド/ランタイムは**チェックイン済みデータを import するだけ**（`egov-caption-snapshot.ts` と同じ静的資産の扱い）。ネットワーク非依存。

これにより「ビルド時に外部APIを叩く」ことの CI/デプロイ不安定リスクを回避しつつ、診断のカバレッジゴールを満たす。

### 4.2 データフロー

```
e-Gov API v2 (GET /api/2/law_data/{lawId} → XML)
  └─ 手動/Cron 実行の取得スクリプト (scripts/etl/egov-fulltext-fetch.ts)
       ├─ XML パース: 条(Article) 単位で articleNum / articleTitle / paragraph 本文を抽出
       ├─ 形式検証: 条番号が取れない/本文空の条はスキップ→ skipped 計上（未確認明記）
       ├─ 正規化: 条番号を当サイト表記（第577条の2）へ・全角/漢数字ゆらぎ吸収（O8-b の kanji-numerals を再利用）
       ├─ diff-only: fetchedAt 以外に差分無ければ書かない
       └─ 出力: web/src/data/laws/<law>.fulltext.generated.ts（LawArticle[] + 出典 + 取得日）
  └─ (チェックイン) → allLawArticles が合流 → search-index / law-search が無改修で全文を引く
```

### 4.3 対象法令（段階投入）

`egov-revisions-fetch.ts` の `TARGET_LAWS`（20法令・lawId 検証済み）を種に、**カバレッジ穴の実害が大きい順**に段階投入:

1. **第1弾（中核3法）**: 安衛法（347AC0000000057）・安衛令（347CO0000000318）・安衛則（347M50002000032）。安衛則の全677条化が最大の穴埋め。
2. **第2弾（主要特別則）**: クレーン則・有機則・特化則・粉じん則・石綿則・酸欠則・電離則・ボイラー則・鉛則・高圧則・ゴンドラ則。
3. **第3弾**: 労基法/労基則・じん肺法・作環測法ほか。

段階投入により、各弾でバンドル影響（§5）を実測しながら進められる。

### 4.4 e-Gov API v2 の実測確認（診断付記より）

- `GET /api/2/laws?law_title=安衛法` → `abbrev`「労安衛法,保安四法,安衛法」で2件（O8-c エイリアスの種にも既利用）。
- 本文取得は `law_data`/`lawdata` エンドポイント（XML 全文）。**API キー不要**＝新規 env なし。

## 5. バンドル肥大対策（診断 G8 の宿題・必須）

全文取込みは条文本文を数千条規模に増やす。現状 `search-index.ts` は全カテゴリをモジュールロード時に構築しており、**全文をそのままクライアントへ載せると初回 JS が肥大 → LCP 悪化・`sw.js` プリキャッシュ肥大**。対策候補（実装時にオーナー/他班と要検討）:

1. **検索インデックスと本文の分離**: 検索に要るのは「条番号＋見出し＋キーワード」。本文全文は**条文詳細を開いた時だけ遅延 fetch**（静的 JSON を条単位 or 法令単位でチャンク化）。`/law-search` のパネルが選択条の本文だけ取りに行く。
2. **法令単位のコード分割**: `<law>.fulltext.generated` を dynamic import。検索ヒット時のみ該当法令チャンクをロード。
3. **PWA プリキャッシュの見直し**: 全文を丸ごとプリキャッシュしない。中核（安衛法/則の見出し＋高頻度条本文）のみプリキャッシュ、残りはオンデマンド＋ランタイムキャッシュ。トンネル/地下の現場優位（G8）は中核維持で担保。

> この分離設計は SEO班（`search-index.ts`／`lib/cross-search`）と、`/law-search` UI 所有班（ux-tools）、PWA/`sw.js` 所有班にまたがる。**横断合意が必要**。

## 6. 品質・整合の担保（テスト設計）

`egov-caption-snapshot.ts` の方式を本文へ拡張:

- **原文照合テスト**: 各法令から数条をサンプリングし、取込本文が e-Gov 公式スナップショット（チェックイン）と**文字一致**（改ざん0）。
- **カバレッジ回帰**: 「安衛則 第577条の2」など既知の未収録条が**取込後にヒット**することを固定（穴が再発しない）。
- **削除条ガード**: 現行法で削除された条がコーパスに混入しないこと（`article-caption-integrity.test.ts` の `"削除"` 規約を踏襲）。
- **出典必須**: 全 `LawArticle` に e-Gov URL＋取得日が付くこと。
- **パフォーマンス**: 検索応答 <200ms（§5 の分離が効いているか）をベンチで固定。
