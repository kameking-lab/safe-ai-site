# 診断報告 02: KY用紙・安全工程打合せ書が「紙の直接操作」になっていない

- 診断日: 2026-07-02（本番 https://www.anzen-ai-portal.jp/ を Playwright で実測）
- 対象: `/ky/paper`（KY用紙）・`/safety-diary`（安全工程打合せ書及び安全衛生指示書）
- 社長の求める姿: **用紙の全貌が一画面で見え、ピンチ/ホイールでズームでき、クリックした場所にその場で入力できる**（全体俯瞰→ズーム→該当欄タップ→インライン入力）
- 制約: A4印刷帳票の正式書式は不可侵（既存の印刷出力は変えない）
- スクリーンショット: `scratchpad\shots\ky\`（PC 1440x900 / スマホ 390x844、top/full/scroll1/zoom60 各種）

---

## 1. 症状の実測（本番）

### 1-1. 「用紙の全貌が一画面で見える」→ 見えない

| 画面 | ページ全長（ビューポート換算） | ファーストビューでの用紙の見え方 |
|---|---|---|
| KY用紙 PC (1440x900) | 3,754px ≒ **4.2画面分** | 用紙はヘッダー行（現場名〜天気）まで。上に結論カード＋4段進行ナビ＋折りたたみ案内が積まれ、用紙の上端が y≈520px から始まる |
| KY用紙 スマホ (390x844) | 6,169css px ≒ **7.3画面分** | **用紙が1pxも見えない**（ヘッダー・表示モードバー・パンくず・ズームピル・結論カード・進行ナビで1画面が埋まる） |
| 打合せ書 PC | 2,885px ≒ 3.2画面分 | ヘッダー欄＋分散入力の説明ボックスまで。各社マトリクスは下 |
| 打合せ書 スマホ | 7,234css px ≒ **8.6画面分** | 用紙見えず。作業日欄に到達するまで約2画面スクロール |

- 実測ショット: `ky-paper-pc-top.png` / `ky-paper-sp-top.png` / `ky-paper-sp-full.png`（縦18,507px の長大フォーム）/ `meeting-sp-full.png`（縦21,702px）

### 1-2. ズームが「俯瞰の道具」になっていない

- ズームは **－/＋ボタンのみ**（60%〜160%、10%刻み）。ピンチ・ホイール・ダブルタップは未実装（`ky-paper` 配下に touch/pinch/wheel/gesture のハンドラは grep 0件）。
- 最小60%にしても（`ky-paper-pc-zoom60.png`）**用紙の1/5程度しか見えない**。CSS `transform: scale()` はレイアウト高さを縮めないため、縮小してもスクロール量は変わらず、右・下に余白が増えるだけ。「縮小＝全体が見える」というユーザーの期待と挙動が食い違う。
- スマホでは60%にしても用紙はファーストビュー外のまま（`ky-paper-sp-zoom60.png`）。ブラウザ標準のピンチはナビごと拡大するだけで用紙操作にならない。
- 「全体を見る（フィット）」ボタンが存在しない。初期表示も全体俯瞰ではなく、いきなり100%の縦積みフォーム。

### 1-3. 画面の「用紙」と印刷される正式書式が別物＝紙のメタファー不成立

- KY: 画面は「枠線つきカードに縦積みのinput/textarea/セレクト」。印刷は `KyPrintSheet` の**黒罫線の表組み（全建協/建災防様式寄せ・確認印枠つき）**。見た目が一致しないため、「いま紙のどの欄を書いているか」が画面から読み取れない。
- 打合せ書はさらに乖離が大きい: 正式書式は **A4横277mmの各社マトリクス（業者×12列の表）** だが、画面は各社ごとの縦積みカード（1社あたり10フィールド）。「表の行を埋める」感覚は皆無で、8.6画面分のフォームを上から順に埋める体験。
- 「今どの欄か」は結論カード（記入のこりN項目）と4段進行ナビで補っているが、これは**紙上の位置**ではなく抽象的なステップ表示。紙で運用してきた職長の頭の中の地図（左上に現場名、真ん中に危険と対策、右下にハンコ）と繋がらない。

### 1-4. 良い資産（すでにあるもの）

- `/ky/paper` は既に「用紙ファースト」思想で、自動保存・音声入力・天気自動取得・作業員マスター・AI提案・承認ロック・クラウド同期・サイネージ連携が揃っている。**欠けているのは見せ方（俯瞰・ズーム・その場入力）だけ**。
- 印刷用レイアウトが **record を受け取る純粋な表示コンポーネント**として独立している（後述）。これが今回の設計の最大の再利用資産。

### 結論（レイアウト定義の持ち方）

- 帳票レイアウトは **HTML/CSS（Tailwindの表組み）で固定実装**。座標データやスキーマ駆動ではない。
- ただし「印刷シート＝`record`だけを受け取る副作用なしのコンポーネント」という構造のおかげで、**印刷シートをそのまま画面キャンバスとして描画し、セルをタップ可能にする**改修が最小差分で成立する。PDF出力は `window.print()`（`@page` 指定＋`print:` クラス）のみで、外部PDFライブラリはなし → 印刷経路を触らなければ書式は不可侵のまま。

## 3. 真因

1. **「用紙ファースト」がフォームの装飾に留まっている**: 画面の"用紙"は枠線を付けた縦積みフォームであり、正式書式（表組み）と別レイアウト。紙の空間記憶が使えない。
2. **ズームが俯瞰装置として設計されていない**: ボタン式10%刻み・最小60%・フィットなし・ピンチ/ホイールなし。さらに `transform: scale` の余白問題でズームアウトの意味が薄い。
3. **用紙の上に「用紙以外」が積まれている**: 結論カード・進行ナビ・折りたたみ案内・通知が用紙より先に表示され、スマホではファーストビューから用紙が消える。
4. **入力が「欄の場所」と切り離されている**: 進行ナビ→アンカースクロールという間接ジャンプで、紙をタップして書く直接操作（direct manipulation）がない。

---

## 4. あるべき姿

1. **開いた瞬間、正式書式のA4用紙が丸ごと1画面に収まって見える**（フィット表示が初期状態。記入済み欄は黒字、未記入欄はうっすら色づけで一目で分かる）。
2. **ピンチ（タッチ）・ホイール（PC）・ボタンの3系統でシームレスにズーム/パン**。ダブルタップ/ダブルクリックで欄へズームイン。「全体」ボタンでいつでも俯瞰へ戻る。
3. **書きたい欄をタップ→その場に入力UIが出る**。スマホは下から出るボトムシート（音声入力つき・「次の欄へ」送り）、PCはセル脇のポップオーバー。入力を確定すると用紙上のその欄に文字が入る（＝印刷とWYSIWYG）。
4. 結論カード・AI提案・承認などの既存機能は**用紙の周辺（上下バーやシート内）に退避**し、用紙が常に主役。
5. 印刷/PDFは現行の `KyPrintSheet`/`MeetingPrintSheet` 経路を**1バイトも変えない**。

## 5. 技術選定と設計

### 5-1. 方式比較

| 方式 | 概要 | 印刷資産再利用 | 入力/IME | ズーム | 実装コスト | 判定 |
|---|---|---|---|---|---|---|
| **(a) 印刷用HTMLを画面キャンバス化＋CSS transformズーム＋欄→フィールドマップのオーバーレイ入力** | `KyPrintSheet` をそのまま画面に描画し、各セルをタップ可能ホットスポットに。入力は contentEditable ではなく、フィールド定義に基づくオーバーレイエディタ | ◎ そのまま流用（画面＝印刷のWYSIWYG） | ◎ 通常の input/textarea（日本語IME・音声入力・既存 `InputWithVoice` 流用可） | ◎ transform: scale＋translate。DOMなのでテキストは常に鮮明 | **S〜M**（レイアウト新造ゼロ） | ✅ **採用** |
| (b) SVG座標マップ＋入力オーバーレイ | 帳票をSVGで再描画し、rect座標で欄を定義 | × 表組みをSVGに再実装（二重管理）。日本語の折り返し・行高可変（riskRows増減）がSVGでは地獄 | ○ オーバーレイなら可 | ◎ viewBoxで自然 | L | ✗ |
| (c) canvas描画 | 帳票をcanvasに描画 | × 全再実装 | × IME・アクセシビリティ・選択・拡大時の再ラスタライズすべて不利 | ○ | L+ | ✗ |

**(a) を推奨する決定的理由**: 両帳票とも「印刷シート＝recordを受け取る純粋コンポーネント」が既に存在するため、(a)なら**画面と印刷が同一ソース**になり、「画面で見た紙がそのまま出てくる」という社長要求の本質（WYSIWYG）を構造的に保証できる。行数が可変（危険行・各社行の追加）でもHTMLテーブルが自動で伸びるので、座標管理が不要。

依存パッケージ追加は不要（ピンチ/ホイールは Pointer Events で自作、~200行）。`react-zoom-pan-pinch` 等の採用も可能だが、CLAUDE.md の「依存パッケージの大幅追加は要確認」に該当するため、まず自作フックで成立させる（うまくいかなければオーナー確認の上で導入検討）。

### 5-2. アーキテクチャ

```
[record（既存: KyInstructionRecordState / MeetingRecord）]
        │  ← 既存の自動保存・クラウド同期・承認・AI提案はそのまま
        ▼
[フィールドマップ ky-fields.ts / meeting-fields.ts]  ← 新規（純データ＋getter/setter）
        ▼
[PaperStage]──ズーム/パン（ピンチ・ホイール・ボタン・フィット・ダブルタップ）
   └─[KyPrintSheet editable]──セルを <EditableCell fieldKey> でラップ
            │ タップ
            ▼
      [FieldEditorSheet]──スマホ=ボトムシート / PC=ポップオーバー
            │ 確定（次の欄へ / 閉じる）
            ▼
        record更新 → 用紙が即時再描画（＝印刷と同じ見た目）
```

#### (1) フィールドマップ（欄→フィールド対応。contentEditableを使わない核心）

```ts
// web/src/lib/ky/paper-fields.ts（新規イメージ）
export type PaperFieldType = "text" | "textarea" | "select" | "date3" | "number" | "workerPicker" | "tags" | "tri";
export type PaperField<R> = {
  key: string;                 // "siteName" / "risk.0.hazard" など
  label: string;               // ボトムシートの見出し（例「危険のポイント（1R）」）
  type: PaperFieldType;
  get: (r: R) => string;
  set: (r: R, v: string) => R; // イミュータブル更新
  options?: { value: string; label: string }[];
  voice?: boolean;             // InputWithVoice を使うか
  next?: string;               // 記入順の次フィールド（「次へ」送り）
};
```

- KYは既存 `computeKyPaperStatus`（未記入判定）とキーを共有し、結論カードの「のこりN項目」→未記入セルのハイライトが自然に接続する。
- 動的行（KY: riskRows、打合せ: contractors/deliveries）は `key` にインデックスを含め、用紙上に「＋行を追加」ホットスポット（印刷時は `print:hidden`）を置く。

#### (2) 印刷シートの editable 化（印刷出力は不変）

- `KyPrintSheet` に `editing?: { onTapField(key): void; activeKey?: string; emptyKeys: Set<string> }` の**省略可能prop**を追加。未指定（＝現行の印刷/プレビュー呼び出し）では**現在と同一のHTMLを出力**（スナップショットテストで担保）。
- 指定時のみ、各 `<td>` の中身を `<EditableCell>` でラップ: `role="button"`・`min-h`確保・未記入はごく薄いアンバー背景＋点線・アクティブは青リング。装飾クラスはすべて `print:` 無効なので印刷に影響なし。
- 参加者・点検○×－・日付などタイプ別に `FieldEditorSheet` 内のエディタを出し分け（作業員マスターのチップ選択UI、Tri トグル等は既存実装を移設）。

#### (3) PaperStage（ズーム/パン 3系統＋フィット）

- 構造: `overflow:hidden` のビューポート div ＋ 内側に `transform: translate(x,y) scale(s)` の用紙。**レイアウト高さ問題を根治**（ビューポートは `height: calc(100dvh - 上下バー)` 固定でページ自体は伸びない）。
- **初期表示＝フィット**: `s = min(vw/paperW, vh/paperH)`（KY: paperW≈820px、打合せ: ≈1,140px 相当）。「⛶ 全体」ボタンでいつでも復帰。
- **ピンチ**: Pointer Events で2点間距離→スケール、中点アンカー。ステージに `touch-action: none`（ページのスクロール/ブラウザピンチと干渉させない。1本指はパン）。
- **ホイール**: ステージ上の `wheel` で Ctrl/⌘＋ホイール（トラックパッドのピンチは ctrlKey付きwheelとして飛んでくるので同経路）＝カーソル位置アンカーのズーム。素のホイールは縦パン。`preventDefault` はステージ内のみ。
- **ボタン**: 既存の −/％/＋ を継続＋「全体」追加。％タップで100%。
- **ダブルタップ/ダブルクリック**: そのセルが読める倍率（例1.2）までズームイン＆センタリング。
- スマホの入力時: セルタップでボトムシートを開く際、対象セルが隠れない位置へ自動パン。iOS の入力フォーカス自動ズームを避けるためエディタのフォントは16px以上。
- 実装: `useZoomPan()` フック（新規 `web/src/lib/ui/use-zoom-pan.ts`、依存追加なし、~200行＋vitest）。

#### (4) 既存機能の再配置（用紙を主役に）

- 結論カード＋進行ナビ → 上部1行のコンパクトバー（「のこり5」タップで最初の未記入セルへズーム）。
- AI提案 → 危険欄セルのエディタ内に「🤖 AI提案」ボタンとして統合（欄の文脈で提案が出る方が自然）。
- 保存/…シート・承認ロック（`inert`）・自動保存・クラウド同期はそのまま流用。

### 5-3. リスクと対策

| リスク | 対策 |
|---|---|
| 印刷シート改修で書式が変わる事故 | `editing` 未指定時のHTMLスナップショットテスト（既存 `ky-print-sheet.test.tsx` に追加）＋印刷プレビューのE2Eスクショ比較 |
| iOS Safari のブラウザピンチ/スクロール干渉 | ステージ `touch-action:none`＋`gesturestart` preventDefault はステージ内限定。ステージ外の通常スクロールは不変 |
| 小倍率でセルが小さくタップ困難 | セルタップは倍率無関係に有効＋タップで自動ズームイン。最低ヒット領域は transform 前の実寸で確保 |
| 打合せ書A4横12列はスマホで文字が小さい | 俯瞰は「地図」と割り切り、タップ→ボトムシートで読む/書く。行単位ズームのダブルタップで補完 |
| 既存ユーザーの慣れ | Phase 1〜2 は `?canvas=1`（またはトグルボタン）で併存→実測OK後に既定化。旧UIは1リリース残す |

### 補足（着手順の理由）
- T1〜T3（Phase 1）だけで社長要求の3要素（全貌・ズーム・タップ入力）がKYヘッダーで実証でき、方式の是非を最小コストで判定できる。
- 印刷不可侵は T2 のスナップショットテストで**最初に**固定するのが肝（以降の全変更が印刷を壊さないことを機械的に保証）。
- 打合せ書（T8〜）は KY で確立した `PaperStage`/`FieldEditorSheet`/フィールドマップ規約をそのまま流用するため、新規設計はフィールド定義のみ。
