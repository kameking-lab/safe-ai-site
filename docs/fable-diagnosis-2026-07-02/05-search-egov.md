# テーマ5: 法令検索「e-Gov超え」残ギャップ棚卸し（2026-07-02）

対象: 本番 https://www.anzen-ai-portal.jp の横断検索（/search・⌘K）・法令全文検索（/law-search）を、e-Gov法令検索（https://laws.e-gov.go.jp/）と同一クエリで並走比較した診断。コード変更なし。実測はPlaywright（本番・e-Gov双方UI駆動）とe-Gov API v2。スクリーンショットは scratchpad\shots\search\（site-search-*.png / egov-n*.png / egov-k*.png ほか25枚）。

## 1. 現状実装の把握（コード読解）

### 1-1. 検索の実体は「2系統＋死蔵1系統」

| 系統 | 入口 | エンジン | 状態 |
|---|---|---|---|
| 横断検索 | /search・⌘K（CommandPalette） | `web/src/lib/search-index.ts` の `searchItems()` | 稼働中（両入口とも同一） |
| 法令全文検索 | /law-search | `web/src/components/law-search-panel.tsx` 内蔵の `normalizeQuery()` | 稼働中 |
| 高機能横断検索 | （なし） | `web/src/lib/cross-search/`（score.ts + build.ts） | **完全な死蔵コード。src配下のどこからもimportされていない** |

### 1-2. 稼働中の /search・⌘K（search-index.ts）のクエリ処理

- 正規化は `web/src/lib/fuzzy-search.ts` の `normalizeSearchText()` のみ: NFKC・長音符統一・小書きかな統一・全半角・小文字化。
- **クエリ全体を1本の文字列として** title/subtitle に `includes` 判定（完全一致100 > 前方一致80 > title含む60 > subtitle含む30）。
- **空白区切りのAND検索なし・シノニム展開なし・漢数字変換なし・条番号パースなし・keywordsフィールド不使用**。
- インデックス: 条文1,065件（curated 50法令、タイトルは「安衛法 第61条」形式=略称+全角条番号）・通達/告示1,069件（→/circulars/[id]深リンク）・判例・化学物質約969件・教育テーマ・事故事例・用語集。モジュールキャッシュで初回のみ構築。

### 1-3. /law-search（law-search-panel.tsx）

- 独自の `kanjiToNum()`（第二十一条→第21条）＋全半角正規化を持ち、**漢数字⇔算用数字はここだけ吸収できる**。
- 検索対象は law+lawShort+articleNum+articleTitle+text+keywords の連結文字列への部分一致。フリーワード欄と条番号欄が分離。
- ただしこちらも**トークナイズなし**（「安衛法61条」は「安衛法 第61条」に不一致で0件）。枝番は「第61条の2」形式のみで「61-2」は不一致。
- 条文カードには e-Gov（法令トップ）リンク・AI要約・引用コピー・お気に入りあり。**本文中の参照（「第30条第1項」「政令で定める」等）は素のテキストでリンクなし**。

### 1-4. 死蔵されている cross-search（皮肉にも一番賢い）

`web/src/lib/cross-search/score.ts` は 空白AND・シノニム展開（`query-expansion.ts`: ヘルメット→保護帽、アスベスト→石綿則 等22ルール）・keywords重み付け・カテゴリ優先タイブレークまで実装済み＆テスト済み（cross-search.test.ts）だが、UIに配線されていない。**今日の /search の敗因の大半は、この実装を使っていないことに起因する。**

### 1-5. 周辺資産

- /law-hierarchy: 法→政令→省令の階層マップ（条文数・関連通達数・e-Govリンク付き）が既にある。
- PWA: public/sw.js が /law-search をプリキャッシュ、_next/static は cache-first → **初回訪問後は条文検索がオフラインでも概ね動く**（e-Govには構造的に不可能）。
- 条文カードのe-Govリンクは `EGOV_LAW_NUMBERS`（25法令の法令ID直書き）で法令トップへ飛ぶのみ。**条アンカー（#Mp-At_61）未使用**。

## 2. 並走比較（本番サイト vs e-Gov、2026-07-02 実測）

計測条件: サイトは /search?q=…（SPAウォーム後、結果描画まで）、e-Govはトップから検索実行→結果件数描画まで。e-Govは新UI（法令名/キーワード/法令番号のラジオ切替、沿革・条文比較・引用元タブあり）。

**(a) 「安衛法61条」→就業制限に着けるか … 引き分け（両者とも生クエリでは0件）**
サイト /search: 0件（1,584ms・初回コールド）。e-Gov 法令名: 0件（1,914ms）、キーワード: 0件（2,886ms）。ただしサイトは「安衛法 第61条」なら完全一致1件・107ms、「61条」でもトップに安衛法第61条（133ms・7件）。e-Govは「安衛法」→略称ヒット→法令ページ→目次スクロールの3ステップ。**復帰しやすさでサイト△勝ち、生クエリはどちらも×＝取りにいける未占地。**

**(b) 「あんえいほう 88条」ひらがな … 引き分け（両者0件）**
サイト0件（100ms）、e-Gov 法令名「あんえいほう」0件（3,523ms）。e-Govは law_title_kana を持ちながらUI検索で引けない。なお「安衛法 88条」もサイト0件（トークナイズ欠如。「第」が挟まるため不一致）。**ひらがな・読みは両者空白＝差別化余地。**

**(c) 「労働安全衛生規則 第563条」正式名称+条番号 … e-Govの勝ち**
サイト /search: **正式名称だと0件**（インデックスのタイトルが略称「安衛則」のため。「安衛則 第563条」なら1件・100ms、足場の作業床に直行）。e-Gov: 法令名「労働安全衛生規則」→1件→法令ページ→目次で第五百六十三条（計10〜20秒だが確実に着く）。**正式名称を打つ真面目なユーザーほどサイトで落ちる、恥ずかしい負け方。**

**(d) 「特化則」略称のみ … サイトの勝ち**
サイト: 23条文が93msで一覧（第2条定義から直読み）。e-Gov: 略称DBで1件ヒット（2,117ms）するが着地は法令トップで、条文を読むにはさらに遷移。**深さと速度で勝ち。** 同型の「クレーン則」ではe-Govは略称未登録で**0件**（1,507ms）、サイトは47件（450ms）＝完勝。

**(e) 「事務所衛生基準規則 気積」 … サイトの勝ち**
サイト: 1件ピンポイント（事務所則 第2条 気積、108ms）。単語「気積」でも2件（安衛則600条・事務所則2条）と精密。e-Gov キーワード: 12件・2,933ms、先頭は船員労働安全衛生規則のノイズ。**精度・速度とも勝ち。**

**(f) 「石綿 事前調査」 … e-Govの勝ち（サイトの最悪ケース）**
サイト /search: **0件**（99ms）。データは存在する（/law-search「事前調査」で石綿則第3条ほか4件）のに、2語クエリをAND分解しないため取り逃す。e-Gov: 4件・2,486ms、石綿則の該当条文スニペット付き。**実装済みデータを検索が殺している代表例。**

**(g) 「クレーン 過負荷」 … e-Govの勝ち（同上）**
サイト /search: 0件。単語「過負荷」なら9件（クレーン則第23条過負荷の制限に到達可）。e-Gov: 14件・2,600ms。敗因は(f)と同じ2語AND欠如。

**(h) 条文からの参照ジャンプ（安衛法20条→関連省令） … e-Govの勝ち**
サイト: 条文本文はプレーンテキスト。keywordsチップも非リンク。e-Govボタンは法令トップ行きで条アンカーなし。e-Gov: 目次アンカー・**引用元タブ**（当該法令を引用する法令の逆引き）・条文比較（沿革）を装備。**構造リンクは現状完敗。**

**(i) 条文に関連する通達・判例の束ね … サイト△勝ち（ただし浅い）**
サイト: /search のカテゴリタブで条文・通達・判例・事故が横並びには出る（例:「就業制限」11件に安衛法61条と就業制限業務従事者講習の告示が同居）。/law-hierarchy は法令単位で関連通達件数リンクあり。ただし**「この条文に紐づく通達・判例」という条文単位の束ねは存在しない**。e-Gov: 通達・判例・解説を一切持たない（構造的に不可能）。**唯一サイトだけが立てる旗だが、まだ旗が小さい。**

**(j) 検索速度 … サイトの圧勝**
サイト /search: ウォーム時 77〜177ms（実測16クエリ中央値約100ms）、コールド初回1.6秒。/law-search: 74〜236ms。e-Gov: 法令名検索1.5〜3.5秒、キーワード検索2.5〜2.9秒（API素叩きでは5〜7秒）、法令本文ページ表示約0.7秒＋目次操作。**体感10〜30倍。加えてPWAでオフライン可。**

**勝敗集計: サイト勝ち4（d,e,i,j）/ e-Gov勝ち4（c,f,g,h）/ 引き分け2（a,b）。**
「速い・略称に強い・束ねの芽がある」が、「2語で引けない・正式名称で引けない・条番号ゆらぎに弱い・参照リンクがない・収録が抄録」で負けている。**負け4つのうち3つ（c,f,g）は既存資産（cross-search・kanjiToNum・law-metadata）の配線だけで返せる。**

## 3. ギャップ棚卸し

### G1. 略称・通称・ひらがな検索 【現状△／e-Govと引き分け〜勝ち】
- 略称（安衛法・特化則・クレーン則・石綿則…）は lawShort として**タイトル側に**あるため強い（e-Govが落とすクレーン則も引ける）。
- ×: 正式名称→条文タイトルに正式名称が無く沈む/落ちる（比較c）。ひらがな読み（あんえいほう）・別通称（労安衛法）は不可。
- 改善設計: `law-metadata.ts` に aliases[]（正式名称・かな読み・別略称・英名）を持たせ、検索側でクエリ内の法令名部分をエイリアス正規化してから照合。e-Gov API の abbrev フィールド（労安衛法,保安四法,安衛法 等）をそのまま種にできる。
- 規模感: 0.5日（50法令×エイリアス表＋照合1関数＋テスト）。

### G2. 条番号ゆらぎ（「61条」「第六十一条」「61-2」「安衛法61条」） 【現状×／e-Govにも無い＝勝ち筋】
- /search は「第61条」の完全表記以外全滅。漢数字は /law-search のみ対応（実測: /law-search「第六十一条」8件、/search 同0件）。「61-2」「61条の2」枝番ゆらぎは両ページとも不可。
- e-Govも「安衛法61条」を法令名・キーワードどちらでも0件にする（実測）。**条番号ダイレクトジャンプはどちらにも無い機能＝先取りできる。**
- 改善設計: クエリパーサ1本。`(法令エイリアス)?\s*(第?([0-9一二三四五六七八九十百千]+)条(の([0-9一二三四五六七八九十]+))?|\d+(-\d+)?条?)` を抽出→ kanjiToNum（既存流用）→ articleNum を「第N条(のM)」へ正規化→ (law, art) 直撃結果を最上位カードで返す。法令指定なしなら候補法令横断で条番号一致を上位表示。
- 規模感: 1日（パーサ＋searchItemsへの合流＋ゆらぎ20パターンのvitest）。

### G3. 複数語AND＋現場語シノニム 【現状×（最重症）／e-Govに負け】
- 「石綿 事前調査」「クレーン 過負荷」「足場 作業床」が全て0件（実測f,g,j4）。データはあるのに検索が殺している。
- **解は書き済み**: cross-search/score.ts（空白AND＋expandQueryシノニム＋keywords重み＋カテゴリ優先）が死蔵。配線替えのみ。
- 規模感: 0.5〜1日（searchItems差し替え or search-indexのSearchItemにkeywords追加してscore.tsを適用。既存テスト資産あり）。**費用対効果が全ギャップ中最大。**

### G4. 条文間参照の自動リンク 【現状×／e-Govに負け（引用元タブ・目次アンカー）】
- 条文本文中の「第30条第1項」「令第6条」「安衛則第36条」が素テキスト。紙のコピーと同じ。
- 改善設計: 表示時linkify。①同一法令内「第N条」→ /law-search?law=…&art=…、②「令第N条」「○○規則第N条」→ law-metadata のエイリアスで解決、③収録外条文は EGOV_LAW_NUMBERS＋e-Gov条アンカー（`https://laws.e-gov.go.jp/law/{lawId}#Mp-At_{N}`、実測で有効）へ外部リンク。全リンクの解決可能性をビルド時テストで担保（幽霊リンクゼロ）。
- 規模感: 2〜3日（linkifier＋解決テーブル＋テスト）。e-Govの引用元タブは「法令間の引用逆引き」で当サイト未着手領域だが、まずは順方向リンクで実利を取る。

### G5. 法令階層ナビ（法→令→則ツリー） 【現状○／e-Govに勝ち】
- /law-hierarchy が既にあり、条文数・関連通達数・e-Govリンク付き。e-Govは五十音・事項別リストのみで体系ツリーなし。
- 残改善: 検索結果・条文カードから階層上の位置（パンくず）へ戻れる導線が無い。小規模（0.5日）。

### G6. 条文↔通達↔判例↔解説の束ね表示 【現状△／e-Govは構造的に不可能＝最大の勝ち筋】
- 現状は「検索結果に同居する」レベル。条文単位の関連付け（安衛法61条→就業制限業務の政令・技能講習の通達・無資格運転の判例・教育資格DB）が無い。
- 改善設計: 条文パーマリンクページ /laws/[law]/[art]（または law-search内の詳細パネル）を作り、(1)G4の参照解決テーブル、(2)通達DB1,069件・判例DB・用語集・教育資格DBへのキーワード/手動マッピングを束ねる。SEO上も「安衛法61条」検索の受け皿になる（現状は条文への安定URLが query string のみ）。
- 規模感: 1〜2週間（テンプレ1本＋関連付けデータ整備。まず安衛法主要100条から段階投入）。

### G7. 収録カバレッジと透明性 【現状×（構造的リスク）／e-Govに負け】
- curated 1,065条文は抄録（安衛則は実際には第677条まである）。収録外の条番号は「該当なし」とだけ出て、**法令上存在しないのか未収録なのか区別できない**（実測: art=第61条の2 →「該当なし」のみ）。誤って「規定がない」と誤読させる安全上のリスク。
- 改善設計: 短期＝0件時に「未収録の可能性。e-Govで確認」ボタン（クエリ・条アンカー引継ぎ）＋収録範囲の明示。中期＝e-Gov API v2（認証不要・XML全文取得可、実測済み）から安衛法・施行令・安衛則＋主要特別則の全条文を取込みビルド時生成（原文一致のサンプル照合テスト付き）。
- 規模感: 短期1日／中期1〜2週間。**中期は外部API取込みのためオーナー確認事項**（CLAUDE.mdルール）。

### G8. 速度・オフライン 【現状○／e-Govに圧勝】
- 実測: サイト77〜177ms vs e-Gov 1,500〜3,500ms。sw.js が /law-search をプリキャッシュ済みでトンネル・地下・山間の現場でも引ける。維持のみ（インデックス全文化時のバンドル肥大に注意。条文全文取込み時は検索ワーカー/遅延ロード設計を再検討）。

## 4. 「e-Gov超え」の勝ち筋3つ（e-Govには構造的にできないこと）

1. **現場語・うろ覚えからの即答**: e-Govは「正式名称・正確な表記の世界」で、略称欠落（クレーン則0件）・かな不可・条番号複合クエリ不可を10年放置している。当サイトは正規化＋エイリアス＋条番号パーサ＋シノニム（死蔵コード含め素材は揃済み）で「安衛法61条」「あんえいほう88条」「ヘルメット 着用義務」を100msで条文に着地させられる。G1+G2+G3で並走比較の負け3本が勝ちに反転し、引き分け2本も勝ちになる。
2. **条文を中心にした束ね（条文↔通達1,069↔判例↔教育資格↔KY/RA機能）**: e-Govは通達・判例・実務解説・資格要件を永遠に持てない（所管外）。「61条を見たら、対象業務の政令・技能講習の通達・無資格運転の送検判例・教育の申込動線まで1画面」は当サイトだけが作れる。G6+G4。SEOの受け皿にもなる。
3. **現場で使える速さと可用性**: 実測10〜30倍の速度差＋PWAオフライン＋AI要約＋引用コピー＋お気に入り。サイネージ・朝礼・現場スマホという利用文脈ごと囲い込む。ただしこの勝ち筋はG7（収録の透明性とe-Govフォールバック）を必ず併設して「速いが穴があるサイト」への信頼転落を防ぐこと。
