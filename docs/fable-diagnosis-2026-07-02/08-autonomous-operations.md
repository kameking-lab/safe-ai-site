# テーマ7: 自律改善の仕組みづくり（社長がプロンプトを投げなくても回り続ける）

診断者: Claude Fable 5（2026-07-02）。実測ベース。

## 症状の実測

**6/14 10:30以降、今日(7/2)までの18日間、機能開発コミットはゼロ。** タグ `sprint-parallel-end-2026-06-14` 以降の185コミットは全てGitHub Actionsの自動データ更新（jma気象・news-feed取込）であり、ローカルの自走ループは1イテレーションも走っていない。真因は以下の複合:

1. **点火系の死（直接原因）**: タスクスケジューラ `safe-ai-loop-runner` は State=Ready で生きているが、Action引数に `-UntilIso "2026-06-12T08:00:00"` が焼き込まれたまま。6/27のログオン時に起動→期限切れ判定→即終了（LastTaskResult=0、正常終了として記録されるため異常に見えない）。トリガはログオン時のみで NextRun も空。**「正常に何もしない」状態が3週間、誰にも見えずに続いた。** メモリ(loop-runner-resurrection)にも「次スプリントで要更新」と書かれていたが、更新する主体（＝次スプリント）自体がこの焼込みで発火しなかった。鶏と卵。
2. **補給系の枯渇**: レーンBACKLOGの未着手は seo=0 / ux-hub=0 / ux-tools=0 / data=2 / ux-records=3（うち1件はPR#558処理済みの残骸）。プロンプトに「3件未満なら自分で補充」ルールはあるが、補充の種（診断・酷評）が供給されないと水増しタスク量産のリスクがあり、6月は人間（社長プロンプト）が種を撒いていた。
3. **二重管理の腐敗**: legacy `BACKLOG.md` に31件の未着手が残るが、C-2-2（ヘッダー検索UI）等、レーン側で出荷済みのものが `[ ]` のまま混在。正本がどれか機械には判定不能。
4. **プロンプトの鮮度切れ**: `loop-prompt.txt` は「あなたはClaude Opus 4.8」宛て・「再開直後の最優先(2026-06-13)」（C-1回収・#95是正、いずれも処理済み）が焼き込まれたまま。次に起動したループは済んだ仕事を再確認する無駄から始まる。
5. **健在な資産**: loop-runner.ps1（レーン別単一インスタンスガード・短時間失敗バックオフ5→10分・-Model引数）、5レーンクローン（safe-ai-lanes/{seo,data,ux-hub,ux-records,ux-tools}、全て6/14 HEADでclean）、レーン別プロンプト5本、CI枠設計（5班×2ジョブ<20枠）。**壊れているのは点火・補給・可視化だけで、エンジンと車体は無傷。**

## 真因（一段深く）

「期限をスケジューラ登録時に焼き込む」設計は、**期限の延長判断を人間の記憶に依存**させる。加えて「正常終了(exit 0)で静かに死ぬ」ため、停止が異常として観測されない。自律システムの死因として最も多い「監視されない正常停止」の典型。

## あるべき姿

社長は結果（マージ済みPRと本番）を見るだけ。点火・補給・点検・報告の4系統が全て自動で、**止まるときは「止まった」と見える形で止まる**。

## 設計

### A. 点火の恒久解: config分離 + launcher（焼込み排除）
- リポジトリに `loop-config.json` を置く（版管理・ループ自身も編集可能）:
  `{ "untilIso": "...", "lanes": [{"name":"data","model":"claude-opus-4-8","enabled":true,"intervalSeconds":150}, ...], "criticEveryDays":7, "lastCriticIso":"..." }`
- 新規 `loop-launcher.ps1`（引数ゼロ）: configを読み、enabledな各レーンの loop-runner.ps1 を `-Lane -Model -UntilIso(configの値)` で起動。既存のレーン別単一インスタンスガードがあるため多重起動は安全（何度呼んでも冪等）。
- スケジューラは **launcher を引数ゼロで登録**し、トリガを「ログオン時」＋「毎日07:00」の2本にする。期限延長は loop-config.json の1行編集のみ。スケジューラ再登録は二度と不要。
- **黙って死なない**: untilIso超過時、launcherは docs/loop-status.md に「期限切れで停止中。延長するには loop-config.json の untilIso を更新」と書いてから終了する。

### B. 補給の自動化: plannerイテレーション
- launcherが各レーンBACKLOGのopen数を数え、3件未満のレーンがあれば、当該レーンの通常イテレーションの前に**plannerプロンプト**（新規 loop-prompt-planner.txt）を1回走らせる: (1)診断ドキュメント（docs/fable-diagnosis-2026-07-02/ 等の提案タスクで未起票のもの）→(2)直近の酷評（docs/site-critique-*/）のS/A残→(3)第三者レビュー方式の自己診断、の優先順で2〜5件補充。水増し禁止・完了条件（実測基準）必須はプロンプトで強制。
- legacy BACKLOG.md は冒頭に「アーカイブ。正本はレーン別BACKLOG」と宣言して凍結（未着手31件は2026-07-02仕分けで再評価済み→各レーンへ移植 or 落選理由を記録）。

### C. 点検の自動化: 週1の外部視点酷評（criticラン）
- launcherが `lastCriticIso` から7日超過を検知したら、通常レーンより先に criticプロンプト（新規 loop-prompt-critic.txt: 忖度なしの第三者ペルソナで本番をPlaywright巡回→docs/site-critique-<date>/ 生成→S/A級を該当レーンBACKLOG冒頭に注入→configのlastCriticIso更新）を1回起動。
- 生成された酷評がplanner（B）の種になる。**診断→起票→実装→出荷→再診断のループが人間なしで閉じる。**

### D. 報告の一元化: loop-status.md
- 各レーンはイテレーション末尾に docs/loop-status.md の自レーン行を更新（最終稼働時刻・直近マージPR・残open数・直近の判断1行）。launcherも起動/停止理由を追記。
- 社長が見るのはこの1ファイル（将来は /site-records のダッシュボードカード化=軸2と統合）。

### E. モデル3層の運転計画
- **〜7/7（Fable枠あり）**: Fableは自動ループに入れない（枠が貴重・1タスクが重い）。社長が1日1回「Fableセッション」を手動起動し、仕分け表の【Fable】タスク（方式確立・監査パイプライン発明）を1〜2本ずつ消化。並行して自動ループは Opus 4.8（dataレーン=法令正確性・重量実装）と Sonnet 5（ux系3レーン=量産横展開）で常時運転。
- **7/8以降**: 【Fable】残タスクは (a)Opusに降格可（タスク側に降格条件を明記済み）か (b)次のFable枠まで凍結、を仕分け表の各タスクに記載の通り処理。configのmodelを書き換えるだけ。
- モデル別の使い分け根拠は docs/fable-diagnosis-2026-07-02/10-task-triage.md 参照。

### F. 安全装置の増強（既存維持＋2点）
- 使用量上限の長期化対応: バックオフ上限を現行10分→30分に延長（上限中に10分毎の空振り起動は無駄。5→10→30分の3段）。
- ループ自身によるconfig編集は untilIso の**短縮方向のみ**許可（延長は社長の1行編集＝人間の意思確認を残す）。

## 規模感

- A launcher+config+スケジューラ再登録: S（2〜4h）— 【Opus】共有インフラで失敗が全レーンを止めるため
- B plannerプロンプト＋BACKLOG.md凍結: S（1〜2h）— 【Opus】
- C criticプロンプト＋launcher連動: S（2h）— 【Opus】（酷評の質はモデルに依存→プロンプトにペルソナ規格を焼く）
- D loop-status.md規約＋各レーンプロンプト追記: S（1h）— 【Sonnet】
- F バックオフ延長: XS（30min）— 【Sonnet】
- loop-prompt.txt群の鮮度是正（Opus宛て文言・済みタスクの除去・モデル非依存化）: S（1h）— 【Sonnet】

依存: A→(B,C,D)。Fは独立。全体でOpus 2〜3イテレーション＋Sonnet 2イテレーション。
