# Daily Review プロンプトテンプレート（スケーラブル版）

このファイルは Cowork の予定済みタスクとして Claude に渡すプロンプトの完成形です。
ペルソナは `web/scripts/personas/` 配下の JSON ファイルに 100 人分定義されており、
`REVIEW_SIZE` 変数で毎回の巡回人数を切り替えられます。

---

## ⚠ 重要：ペルソナの役割についての大前提

これらのペルソナは全員、**サイトの繁栄を願う辛口クリティック**である。
彼らの本音はサイト改善の**最重要シグナル**である。

忖度・遠慮・気遣いは一切なし。
「いい点もある」みたいな中途半端な評価はしない。
ダメなものはダメ、使えないものは使えないと言い切る。
改善案も具体的に提示する。
ただし人格攻撃や感情的な批判はしない、あくまで事実ベース。

**PM はこの本音を薄めず、そのまま集約してレポートに残すこと。**
丸めた・無難な評価を書き残したら、このレビュー制度の意味が失われる。

---

## REVIEW_SIZE 変数

```
REVIEW_SIZE = 5
```

**利用可能な値**: `5` / `10` / `20` / `50` / `100`

| サイズ | 用途 | 推奨モデル |
|-------|------|----------|
| 5     | 毎日の軽量巡回（Daily） | Sonnet 4.6 |
| 10    | 隔日のやや広めレビュー   | Sonnet 4.6 |
| 20    | 週次レビュー（Weekly）  | Sonnet 4.6 |
| 50    | 月次レビュー（Monthly） | Opus 4.6 1M 推奨 |
| 100   | 四半期レビュー（Quarterly） | Opus 4.6 1M 必須、複数セッション分割可 |

## グループ割当ルール（デフォルト）

| 曜日 | グループ |
|------|----------|
| 月   | A        |
| 火   | B        |
| 水   | A        |
| 木   | B        |
| 金   | A        |
| 土   | B        |
| 日   | A        |

`REVIEW_SIZE` に応じて、以下のロジックでペルソナを選出する：

- `REVIEW_SIZE <= 50` のとき: **当日グループ（A or B）からランダムに REVIEW_SIZE 人を抽出**
- `REVIEW_SIZE == 100` のとき: **全ペルソナを参加させる（グループ指定は無視）**

## ワークスペース情報

- ブランチ戦略: 作業ブランチに commit、最後に PR 作成（または main 直 push は運用者指示に従う）
- ルールファイル: `web/scripts/daily-review.md` を最初に必ず読むこと

## Step 1: 準備
1. `cd web && npm run build` でビルドが通ることを確認
2. `git pull` で最新化
3. `web/scripts/daily-review.md` でルール確認
4. `web/scripts/personas/` 配下の以下 5 ファイルをすべて読み込む:
   - persona-001-010.json
   - persona-011-030.json
   - persona-031-050.json
   - persona-051-070.json
   - persona-071-100.json
   合計 100 人分のペルソナが手元に揃うこと。

## Step 2: ペルソナ選出
- 今日の曜日からグループを決める:
  - 月・水・金・日 → グループ A
  - 火・木・土 → グループ B
- 選出ロジック:
  - REVIEW_SIZE <= 50 なら、当日グループから REVIEW_SIZE 人をランダム抽出（業種・職種・性別が偏らないよう配慮）
  - REVIEW_SIZE == 100 なら、全 100 人（A50 + B50）を参加させる
- 選出したペルソナの id・name・group を一覧にしてレポートの冒頭に列挙する

## Step 3: ローカル確認
- `cd web && npm run dev` でローカルサーバー起動（可能な範囲で）
- サイトの主要ページを巡回する（URL 一覧は `web/scripts/daily-review.md` 参照）
- スマホ（375px）/ PC（1440px）両方で確認

## Step 4: ペルソナ個別レビュー（辛口・本音）
選出した各ペルソナに対し、以下を必ず埋める：
- 使ってみて「本当にダメだと思った点」を最低 2 件
- 「競合・一次ソースで十分」と感じる箇所を最低 1 件（あれば）
- 改善案（具体的・実装レベル、「何となく」禁止）

本音を丸めない。遠慮・忖度なし。
ただし人格攻撃・感情的批判は禁止、事実ベースで書く。

## Step 5: PM 会議（統合）
PM ロール（ペルソナではない、司会役）として：
1. 全員の指摘を集計、重複は統合、対立はフラグを立てる
2. 「深刻度（High / Mid / Low）」と「工数（S / M / L）」でマトリクス化
3. その場で修正してよい「軽微な修正」を特定（下記基準）
4. オーナー判断が必要な「大きな提案」を特定
5. 総合スコアを算出（5 点満点、採点基準は `daily-review.md` 参照）

## Step 6: 軽微な修正を実施
対象の判断は `daily-review.md` の「修正可否の判断基準」に従う。
修正後、必ず `npm run build && npm run lint` でチェック。
コミットメッセージ例: `fix: [修正内容の概要]`

## Step 7: レポート保存
レポートは以下のパスに保存する:
  `web/scripts/daily-review-{{TODAY}}.md`

レポート構成:
```markdown
# 🗓 Daily Review - {{TODAY}}（REVIEW_SIZE={{REVIEW_SIZE}} / グループ: {{GROUP}}）

## 👥 参加ペルソナ（{{REVIEW_SIZE}}人）
<!-- id / group / name / role を列挙 -->

## 🔥 ペルソナ別 本音レビュー（辛口）
<!-- 各ペルソナごとに見出しを立て、事実ベースで辛口に -->
### ID {{id}}: {{name}}（{{role}}）
- **ダメだと思った点 1**: ...
- **ダメだと思った点 2**: ...
- **競合・一次ソースで十分な箇所**: ...
- **改善案**: ...

## ✅ 本日修正した項目
<!-- コミットハッシュと修正内容 -->

## 💡 改善提案（オーナー判断待ち）
### 提案N: タイトル
- 提案者: 誰
- 深刻度: High / Mid / Low
- 工数: S（1h以内）/ M（半日）/ L（1日以上）
- 優先度: 高 / 中 / 低
- 理由（事実ベース・複数ペルソナの共通見解なら強調）

## 📊 総合スコア: X.X/5.0
採点基準: ユーザビリティ + コンテンツ品質 + 信頼性 + SEO/技術品質 + モバイル対応 の 5 点満点

## 🏆 今日のMVPペルソナ
- ID / name / 理由（最も鋭い指摘をした人）

## 🧊 冷静な総括（PMの本音）
1-3 段落で、今日のサイトの姿を率直に書く。
「改善の方向性がぶれていないか」「競合と比べてどうか」を明記。
```

## 実行後の確認事項

1. `web/scripts/daily-review-{{TODAY}}.md` が作成されている
2. レビューが **薄まっていない**（各ペルソナで最低 2 件のダメ出しがある）
3. 軽微修正のコミットがあれば build/lint が通っている
4. PM の「冷静な総括」が書かれている
5. 総合スコアが数値として記録されている
