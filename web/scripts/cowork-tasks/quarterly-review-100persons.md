# Quarterly Review - 100人（四半期レビュー）

> **必須モデル: Opus 4.6 1M（複数セッション分割可）**
>
> 100 ペルソナは最大の負荷。Opus 4.6 の 1M context なら 1 セッションで完結できる。
> 文脈が逼迫する場合は、A グループ 50 人 → B グループ 50 人の **2 セッション分割** を行い、
> 最後に別セッションで両者の結果を **統合** するのも可。
>
> **NG**: Sonnet 4.6 だけで 100 人を 1 セッションで処理する（品質が劇的に下がる）。

このファイルは Cowork の予定済みタスクとして Claude に渡すプロンプトの完成形です。
ペルソナは `web/scripts/personas/` 配下の JSON ファイルに 100 人分定義されており、
本タスクは **全員参加** の全社総決起集会型レビューです。

---

## ⚠ 重要：ペルソナの役割についての大前提

これらのペルソナは全員、**サイトの繁栄を願う辛口クリティック**である。
彼らの本音はサイト改善の**最重要シグナル**である。

忖度・遠慮・気遣いは一切なし。
「いい点もある」みたいな中途半端な評価はしない。
ダメなものはダメ、使えないものは使えないと言い切る。
改善案も具体的に提示する。
ただし人格攻撃や感情的な批判はしない、あくまで事実ベース。

**PM はこの本音を薄めず、そのまま集約してレポートに残すこと。**
丸めた・無難な評価を書き残したら、このレビュー制度の意味が失われる。

## REVIEW_SIZE 変数

```
REVIEW_SIZE = 100
```

## グループ割当ルール（四半期では無視）

`REVIEW_SIZE = 100` のとき: **全 100 人（A 50 + B 50）が参加。曜日・グループルールは無視。**

## 大前提（必ず読む）
これらのペルソナは全員、サイトの繁栄を願う辛口クリティックである。
彼らの本音はサイト改善の最重要シグナルである。
丸めない・薄めない。ダメなものはダメと書き残すのが PM の責務。

四半期レビューは最重要。ここでの指摘は次の 3 ヶ月間の開発方向性を決める。
**各ペルソナの辛口コメント（ダメ出し1件分）は原文保持必須**。

## モデルと分割戦略
- Opus 4.6 1M で 1 セッション完結が理想
- 文脈が逼迫したら:
  - セッション 1: グループ A（id 奇数 50人）のレビュー → 中間レポート
  - セッション 2: グループ B（id 偶数 50人）のレビュー → 中間レポート
  - セッション 3: 両者を統合 → 最終レポート
- Sonnet 4.6 のみで 100 人を 1 セッションで処理することは避ける

## ワークスペース情報
- リポジトリ: C:\Users\kanet\OneDrive\ドキュメント\safe-ai-site
- ブランチ戦略: 作業ブランチに commit、最後に PR 作成（または main 直 push は運用者指示に従う）
- ルールファイル: `web/scripts/daily-review.md` を最初に必ず読むこと

## Step 1: 準備
1. `cd web && npm run build` でビルドが通ることを確認
2. `git pull` で最新化
3. `web/scripts/daily-review.md` でルール確認
4. `web/scripts/personas/` 配下の以下 5 ファイルをすべて読み込む:
   - persona-001-010.json
   - persona-011-030.json
   - persona-031-050.json
   - persona-051-070.json
   - persona-071-100.json
   合計 100 人分のペルソナが手元に揃うこと。

## Step 2: ペルソナ選出
- 全 100 人を参加対象とする（曜日・グループ無視）

## Step 3: ローカル確認
- `cd web && npm run dev` でローカルサーバー起動（可能な範囲で）
- サイトの全ページを巡回する（URL 一覧は `web/scripts/daily-review.md` 参照）
- スマホ（375px）/ タブレット（768px）/ PC（1440px）の 3 解像度で確認
- 過去 3 ヶ月の変化、4 半期 KPI の達成度、競合動向まで視野に入れる

## Step 4: ペルソナ個別レビュー（辛口・本音）
100 人全員に対し、以下を必ず埋める：
- 使ってみて「本当にダメだと思った点」を最低 2 件
- 「競合・一次ソースで十分」と感じる箇所を最低 1 件（あれば）
- 改善案（具体的・実装レベル、「何となく」禁止）

**PM の責務**:
- 全員の「ダメだと思った点 1」を原文で残す（要約禁止）
- それ以外は要約可だが、トーンを殺さない

## Step 5: PM 会議（統合・戦略会議）
PM ロール（ペルソナではない、司会役）として：
1. 全 100 人の指摘を集計、重複統合、対立フラグ
2. **クロス集計**:
   - 業種 × 機能（例: 建設業は KY を評価するが Eラーニング不満、など）
   - 属性 × 機能（例: 外国人は翻訳不足、女性は装備情報不足）
   - デバイス × 機能（例: スマホユーザーは読み込み速度、PCユーザーは印刷）
3. **競合との戦略ギャップ分析**
4. **四半期 KPI 達成度評価**（ユーザビリティ / 品質 / 信頼性 / SEO / モバイル）
5. **翌四半期の戦略提言**: トップ 5 の優先施策

## Step 6: 軽微な修正を実施
対象の判断は `daily-review.md` の「修正可否の判断基準」に従う。
修正後、必ず `npm run build && npm run lint` でチェック。

## Step 9: Push / PR
- 修正コミットがある場合:
  - 作業ブランチに push
  - PR を作成（`gh pr create`）、四半期レビューは必ず PR 化
- 修正がなくても、生成したレポートはコミット・push すること

## Step 10（分割実行時のみ）: 統合セッション
セッション分割した場合、最後に別セッションで以下を行う:
- 両セッションの中間レポートを入力として読み込む
- 上記のレポートフォーマットに沿って最終統合レポートを生成
- クロス集計（業種×機能など）は統合セッションで実施する
